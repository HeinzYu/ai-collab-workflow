# ai-collab-workflow 启动语使用指南

> **基于**: `ai-collab-workflow` v1.9 优化方案
> **制定日期**: 2026-07-02
> **适用范围**: 所有用户（从初学者到高级开发者）

---

## 📖 使用说明

本指南列出了 `ai-collab-workflow` 技能的所有启动语，按**场景分类**。所有指令均采用**自然语言对话风格**，无需记忆任何特殊语法或命令行参数。

**核心原则**: 像日常对话一样表达你的需求即可。

---

## 📋 场景分类总览

| 类别 | 编号 | 启动语数量 | 适用阶段 |
|------|------|-----------|---------|
| 一、项目初始化 | §1 | 4 条 | 全新项目创建 |
| 二、已有项目集成 | §2 | 3 条 | 已有代码库嵌入工作流 |
| 三、模块开发管理 | §3 | 4 条 | 日常功能开发与续接 |
| 四、项目查询与管理 | §4 | 2 条 | 进度查看与状态监控 |
| 五、项目维护管理 | §5 | 3 条 | 定期文档维护与归档 |
| 六、会话管理 | §6 | 2 条 | 每日开发开始与结束 |
| 七、合规检查管理 | §7 | 4 条 | 项目文档质量检查与修复 |
| 八、自动触发场景 | §8 | 6 种 | 系统自动执行（无需用户输入） |

## §1 项目初始化类启动语

> **适用场景**: 全新项目的初始设置，根据项目规模和类型选择不同模式。

### 1.1 Full 模式（完整商业项目）

**触发语**:
- "这是一个完整的 Apple App 项目，包含 [功能]...等功能"
- "这是一个 Web 项目，包含 [功能]...等功能"

**适用场景**: 全新商业级应用（iOS/macOS/Web），包含多个功能模块。

**系统动作**:
1. 自动检测目录状态（空目录 → Full 模式；已有代码 → 降级为 Merge 模式）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 生成完整的目录结构（AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md 等）
4. 初始化所有模块占位符（PRD/、ARCHITECTURE/、TESTING/、DEPLOYMENT/ 等）
5. 初始化所有补充章节（S01-S07）

**示例**:
```
"这是一个完整的 Apple App 项目，包含登录、首页、设置等功能"
"这是一个 Web 项目，包含用户系统、商品管理、支付功能等"
```

---

### 1.2 Small 模式（小型工具项目）

**触发语**:
- "这是一个简单的工具 App，只需要 [功能1] 和 [功能2]"

**适用场景**: 小型工具类应用，只需核心模块。

**系统动作**:
1. 自动检测目录状态（空目录 → Small 模式；已有代码 → 降级为 Merge 模式）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 生成核心文件 + 选中的模块占位符
4. 跳过 TESTING/、DEPLOYMENT/（简化流程）
5. 在 PROJECT_CONTEXT.md 中标记 `scope: "small"`

**示例**:
```
"这是一个简单的工具 App，只需要登录和首页功能"
"这是一个小型管理后台，只需要用户管理和数据导出功能"
```

---

### 1.3 MVP 模式（最小可行产品）

**触发语**:
- "这是一个原型验证，只需要 [一个功能]"

**适用场景**: 最小可行产品验证，只生成对应 PRD 占位符。

**系统动作**:
1. 自动检测目录状态（空目录 → MVP 模式；已有代码 → 降级为 Merge 模式）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 仅生成核心文件 + 对应的 PRD 占位符
4. 跳过所有其他模块
5. 在 PROJECT_CONTEXT.md 中标记 `scope: "mvp"`

**示例**:
```
"这是一个原型验证，只需要一个页面"
"这是一个 MVP 项目，只需要用户注册功能"
```

## §2 已有项目集成类启动语

> **适用场景**: 已有代码库，需嵌入标准文档框架。系统会自动检测目录状态并降级为合适模式。

### 2.1 Merge 模式（标准集成）

**触发语**:
- "帮我给这个项目加上协同工作流"
- "为这个项目初始化协作开发模板"

**适用场景**: 已有代码库，需嵌入标准文档框架。

**系统动作**:
1. 运行全局前置检测（检查目录中是否有现有代码）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 列出已有文件清单（inventory）
4. 展示合并计划（新增哪些文件、如何处理已有文件）
5. 等待用户确认后执行

**输出示例**:
```
检测到这是一个已有项目。在开始工作前，已与您确认模型信息（见 MODEL_CONFIG.md §0）。
我建议以下操作：
- 保留现有源码文件（不修改）
- 新增以下框架文件：AGENTS.md, PROJECT_CONTEXT.md, ...
- 以下文件已存在，建议：[跳过 / 合并 / 备份]
- 请问如何继续？
```

---

### 2.2 Shell Mode（外壳模式，v1.9 新增）

**触发语**:
- "帮我给这个项目加上协同工作流" *(系统自动检测后提示)*

**适用场景**: 项目已有功能等价文档（如 `AI_CONTEXT_MAP.md`、`AI_PROGRESS_MAP.md`），只需补充流程层文件。

**系统动作**:
1. 运行全局前置检测 + 交互式模型对话（同 Merge 模式）
2. **功能等价检测**：扫描项目自有文档，判断是否与标准文件功能等价
3. 如果检测到 ≥2 个功能等价物 → 自动进入 Shell Mode

**输出示例**:
```
检测到此项目已有完善的文档体系：
  - AI_CONTEXT_MAP.md → 替代 PROJECT_CONTEXT.md（任务路由 14 类 + 行号精度）
  - AI_PROGRESS_MAP.md → 替代进度追踪（23+ 模块 + N1-N35 编号）
  - 技术文件框架/ → 替代 ARCHITECTURE/（E2-E6 工程规格）

我建议用"外壳模式"，只补充流程层文件：
  ✓ 新增: AGENTS.md, MODEL_CONFIG.md, REFERENCES.md, DEVLOG/, CONTINUE/
  ✗ 跳过: PROJECT_CONTEXT.md, PRD/, ARCHITECTURE/, TESTING/, DEPLOYMENT/

请问是否接受？
```

**功能等价映射表**:

| 标准文件 | 项目自有文档（功能等价物） | 检测信号 |
|---------|------------------------|---------|
| `PROJECT_CONTEXT.md` | `AI_CONTEXT_MAP.md` | 包含 "任务路由" + P0/P1/P2 标注 |
| `PROJECT_CONTEXT.md`（进度部分） | `AI_PROGRESS_MAP.md` | 包含模块完成度百分比 + N1-Nxx 编号 |
| `ARCHITECTURE/` | `技术文件框架/` | 包含 E2/E3/E4/E5/E6 命名的工程规格文件 |
| `PRD/` | `PRD_` 前缀的文件 | 包含功能需求描述 |
| `TESTING/` | 代码中的测试文件 | 不创建新目录，在 AGENTS.md 中标注 |

## §3 模块开发管理类启动语

> **适用场景**: 日常功能开发与续接，适用于已有项目的持续开发阶段。

### 3.1 生成模块详细 PRD

**触发语**:
- "生成 [模块名] 功能的详细 PRD"

**适用场景**: 已有项目中需补充特定功能的需求文档。

**系统动作**:
1. 在 PRD/ 目录下生成对应模块的详细需求文档
2. 包含功能描述、用户故事、验收标准

**示例**:
```
"生成用户登录功能的详细 PRD"
"生成商品管理模块的详细 PRD"
```

---

### 3.2 生成模块总结（CONTINUE/）

**触发语**:
- "生成 [模块名] 功能的总结"

**适用场景**: 模块开发完成后，自动生成总结文档。

**系统动作**:
1. 在 CONTINUE/ 目录下生成模块总结文档
2. 记录关键决策、技术选型、待解决问题

**示例**:
```
"生成用户登录功能的总结"
"生成支付模块的总结"
```

---

### 3.3 续接已有功能

**触发语**:
- "继续开发 [模块名]，参考之前的总结"

**适用场景**: 中断后继续开发，自动加载相关 CONTINUE/ 摘要。

**系统动作**:
1. 查找对应模块的 CONTINUE/ 总结文档
2. 加载相关决策日志和技术上下文
3. 从上次中断处继续开发

**示例**:
```
"继续开发用户登录功能，参考之前的总结"
"继续开发支付模块，参考上次的总结"
```

---

### 3.4 每日开发启动

**触发语**:
- "开始今天的开发，当前模块是 [模块名]"

**适用场景**: 每日开发开始时指定当前工作模块。

**系统动作**:
1. 加载永久层文件（AGENTS.md、MODEL_CONFIG.md、动态检测的项目自有文档）
2. 加载对应模块的 PRD 和 CONTINUE/ 摘要
3. 显示当前项目状态摘要

**示例**:
```
"开始今天的开发，当前模块是用户登录"
"开始今天的开发，当前模块是支付功能"
```

---

## §5 项目维护管理类启动语

> **适用场景**: 定期文档维护与归档，适用于项目运行一段时间后的常规维护。

### 5.1 执行周维护

**触发语**:
- "执行本周项目维护"

**适用场景**: 每周自动检查 DEVLOG/、更新进度、生成周报。

**系统动作**:
1. 检查最近 7 天的 DEVLOG/ 日志
2. 更新 PROJECT_CONTEXT.md 中的模块进度
3. 生成周报摘要（写入 DEVLOG/weekly-YYYY-Www.md）

**输出示例**:
```
本周项目维护完成:
- 新增开发日志: 5 条
- 更新的模块: 首页展示 (75% → 80%)
- 风险告警: 1 条（已处理）

周报已生成: DEVLOG/weekly-2026-W27.md
```

---

### 5.2 执行月归档

**触发语**:
- "执行本月项目归档"

**适用场景**: 生成月度总结，归档到 SUPPLEMENTARY/ 和 ARCHIVE/。

**系统动作**:
1. 检查最近 30 天的所有日志和文档变更
2. 生成月度总结报告（写入 SUPPLEMENTARY/monthly-YYYY-MM.md）
3. 更新 ARCHIVE/ 目录

**输出示例**:
```
本月项目归档完成:
- 总开发日志: 20 条
- 完成的模块: 用户登录 (100%)
- 进行中的模块: 首页展示 (80%)

月报已生成: SUPPLEMENTARY/monthly-2026-07.md
```

---

### 5.3 执行季度审计

**触发语**:
- "执行季度项目审计"

**适用场景**: 全面审查项目健康度、合规评分趋势、已验证约束提取。

**系统动作**:
1. 检查最近 90 天的所有日志和文档变更
2. 全面审查项目健康度（模块进度 + 合规评分 + 风险告警）
3. 提取已验证约束（经过多次对话验证的设计决策）并写入 AGENTS.md
4. 生成季度审计报告

**输出示例**:
```
季度项目审计完成:
- 总开发日志: 60 条
- 完成的模块: 2/3 (用户登录 ✅, 首页展示 ✅)
- 合规评分趋势: ↑ 从 65% 提升到 78%

已验证约束提取:
- 技术选型: 使用 SwiftUI 作为前端框架 (经 5 次对话验证)
- 架构决策: 采用 MVVM 模式 (经 8 次对话验证)

审计报告已生成: SUPPLEMENTARY/quarterly-2026-Q3.md
```

## §6 会话管理类启动语

> **适用场景**: 每日开发开始与结束，确保每次对话的状态连续性。

### 6.1 会话结束（保存进度）

**触发语**:
- "今天先这样，保存进度"
- "执行工作结束清单"

**适用场景**: 每日开发结束时，自动更新 DEVLOG/、保存进度。

**系统动作**:
1. 更新对应模块的 PRD（如有需求变更）
2. 追加 DEVLOG/ 日志（记录本次会话内容）
3. 如模块完成，生成 CONTINUE/ 摘要
4. 更新 PROJECT_CONTEXT.md 中的模块状态
5. 如做出重要决策，生成决策日志
6. 检查上下文使用量（>80% 时建议开启新对话）
7. 向用户展示"会话工作总结"，确认状态

**输出示例**:
```
今日工作结束清单完成:
- 本次会话新增日志: 3 条
- 更新的模块: 首页展示 (75% → 80%)
- 重要决策: 采用 SwiftUI 作为前端框架 (已记录到决策日志)

会话工作总结:
- 开始时间: 2026-07-02 14:00
- 结束时间: 2026-07-02 15:30
- 完成工作: 首页展示模块的 UI 组件开发

下次会话可从"继续开发首页展示功能"开始。
```

---

### 6.2 新会话启动（v1.9 新增）

**触发语**:
- "开始新的对话"

**适用场景**: 开启全新会话，自动加载永久层文件。

**系统动作**:
1. 加载永久层文件（根据 Context Budget 自适应）:
   - P0 档位（所有用户）: AGENTS.md + MODEL_CONFIG.md + [项目上下文文件]
   - T1 用户（≤16GB）: 仅 P0 文件（预算 ≤8KB）
   - T2 用户（17-32GB）: P0 + 首个 P1 文件（预算 ≤15KB）
   - T3+ 用户: 可加载更多 P1/P2 文件（预算逐级递增）
2. 检查维护日期（如周/月维护已过期，提醒用户）
3. 显示项目状态摘要

**档位与预算对照表**:

| 你的硬件档位 | 上下文窗口 | 永久层预算上限 | 预计可加载文件数 |
|-------------|-----------|--------------|----------------|
| T1 (≤16GB) | 16K tokens | ≤8KB | 2-3 个 (P0 only) |
| T2 (17-32GB) | 24K tokens | ≤15KB | 3-4 个 (P0 + 部分 P1) |
| T3 (33-48GB) | 32K tokens | ≤20KB | 4-5 个 (P0 + 全部 P1) |
| T4 (49-64GB) | 48K tokens | ≤30KB | 5-7 个 (P0 + P1 + 部分 P2) |
| T5 (≥65GB) | 64K+ tokens | ≤50KB | 7+ 个 (全量加载) |

## §7 合规检查管理类启动语（v1.9 新增）

> **适用场景**: 项目文档质量检查与修复，所有指令均采用自然语言对话风格。

### 7.1 检查项目合规状态

**触发语**:
- "检查一下项目的合规情况"
- "当前项目符合标准框架吗？"

**适用场景**: 查看当前项目标准框架文件的合规评分（0-100%）。

**系统动作**:
1. 运行 `check-compliance.sh`
2. 显示总评分（0-100%）
3. 列出缺失或不符合标准的文件

**输出示例**:
```
项目合规检查:
━━━━━━━━━━━━━━━━━━━━━━━
总评分: 78/100

核心文件检查:
- AGENTS.md: ✅ 存在 (内容完整)
- PROJECT_CONTEXT.md: ✅ 存在 (内容完整)
- MODEL_CONFIG.md: ✅ 存在 (内容完整)

Context 生命周期检查:
- Permanent 层: ✅ 正常
- Working 层: ⚠️ 需更新 (3 个文件过期)

风险告警:
- [低] DEPLOYMENT/ 目录为空 (可选模块，不影响)
```

---

### 7.2 查看详细合规报告

**触发语**:
- "显示详细的合规检查报告"
- "帮我看看项目文档缺哪些部分"

**适用场景**: 分阶段（核心文件 / Context 生命周期 / 硬件自适应 / 维护机制）展示详细报告。

**系统动作**:
1. 运行 `check-compliance.sh --detail`
2. 分阶段展示详细报告

**输出示例**:
```
详细合规检查报告:
━━━━━━━━━━━━━━━━━━━━━━━

【第一阶段】核心文件存在性
- AGENTS.md: ✅ 存在 (内容完整)
- PROJECT_CONTEXT.md: ✅ 存在 (内容完整)
- MODEL_CONFIG.md: ✅ 存在 (内容完整)
- REFERENCES.md: ✅ 存在 (内容完整)

【第二阶段】Context 生命周期检查
- Permanent 层: ✅ 正常 (3 个文件)
- Working 层: ⚠️ 需更新 (2 个文件过期 >7 天)
- Ephemeral 层: ✅ 正常

【第三阶段】硬件自适应检查
- 当前档位: T3 (32K tokens)
- Context Budget: ✅ 正常 (18KB/20KB)

【第四阶段】维护机制检查
- 周维护: ✅ 上次运行于 2 天前
- 月归档: ⚠️ 上次运行于 35 天前 (建议执行)
```

---

### 7.3 自动修复合规缺口

**触发语**:

## 🔧 附录：技术细节（开发者参考）

### A. Context Budget 自适应算法

初始化时执行以下调度算法：
1. 检测所有项目自有核心文件（AI_CONTEXT_MAP.md, AI_PROGRESS_MAP.md 等）
2. 估算每个文件的 token 消耗（按 KB × 1.5 安全系数）
3. 根据 T1-T5 档位确定永久层预算上限
4. 按优先级（P0 > P1 > P2）填充永久层，直到预算耗尽
5. 超出预算的文件自动降级为 "Stable Layer"（按需加载，非永久层）

### B. Shell Mode 功能等价检测

当检测到 ≥2 个功能等价物时，自动进入 Shell Mode：
- 仅创建流程层文件（AGENTS.md, MODEL_CONFIG.md, REFERENCES.md, DEVLOG/, CONTINUE/）
- 跳过标准框架文件（PROJECT_CONTEXT.md, PRD/, ARCHITECTURE/ 等）
- AGENTS.md §3 (Always Load) 动态列出项目自有文件

### C. 合规检查脚本集成

- `check-compliance.sh` 支持四阶段检查（核心文件 / Context 生命周期 / 硬件自适应 / 维护机制）
- 评分范围: 0-100%
- 趋势追踪: 写入 `.compliance_history`，支持 `--trend` 查看历史
