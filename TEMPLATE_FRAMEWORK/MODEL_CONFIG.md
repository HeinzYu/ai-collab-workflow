# MODEL_CONFIG.md — 本地模型配置

> 此文件为始终详细的核心文档，定义 oMLX 托管的本地模型配置、对话管理策略与大文件处理优化。每次新项目初始化时由 TRAE 根据实际部署情况填充。

---

## 1. oMLX 服务配置

### 1.1 连接信息
- **服务地址：** http://localhost:8080（默认 oMLX 本地地址，可根据实际部署修改）
- **认证方式：** 无认证（本地运行，外网部署时建议添加 API Key）
- **认证凭证存储位置：** `.trae/models/credentials`（不提交到版本控制）

### 1.2 API 端点
| 功能 | 端点路径 | 说明 |
|------|----------|------|
| 文本生成 | `/v1/completions` | 通用文本生成 |
| 对话交互 | `/v1/chat/completions` | 多轮对话 |
| 模型列表 | `/v1/models` | 获取可用模型 |

### 1.3 连接参数
- **超时时间：** 60 秒（单次请求）
- **重试次数：** 3 次（指数退避：1s, 2s, 4s）
- **并发限制：** 2 个并发请求（M5 Pro 48GB 建议值，可根据实际硬件调整）

## 3. 模型选择策略

### 3.1 自动选择规则

| 任务类型 | 推荐模型 | 理由 |
|----------|----------|------|
| 代码生成（<500行） | Gemma | 响应快，适合小任务 |
| 代码生成（>500行） | Qwen | 需要更大上下文窗口 |
| 文档摘要（<2000字） | Gemma | 轻量任务，快速响应 |
| 文档摘要（>2000字） | Qwen | 需要处理长文本 |
| 大文件（doc/PDF）处理 | Qwen | 需要长上下文窗口 |
| Figma 设计稿拆分 | Qwen | 需要处理多页面内容 |
| 简单问答/查询 | Gemma | 快速响应，节省资源 |

### 3.2 降级策略
当推荐模型不可用时：
1. 尝试切换到备用模型（Gemma ↔ Qwen）
2. 若两个模型都不可用，记录错误到 `DEVLOG/` 并通知用户

## 4. 对话管理策略

### 4.1 上下文窗口使用规范

| 模型 | 最大 Tokens | 安全上限（80%） | 建议单次对话输入 |
|------|-------------|-----------------|------------------|
| Gemma | 8192 | 60% | 核心文件 + 当前模块 PRD |
| Qwen | 32768 | 80% | 核心文件 + 当前模块 PRD + 相关总结 |

### 4.2 会话生命周期管理

#### 会话开始
1. TRAE 读取 `AGENTS.md`、`PROJECT_CONTEXT.md`、`MODEL_CONFIG.md`
2. 读取当前模块的 `PRD/<module>.md`（如正在开发该模块）
3. 读取相关 `CONTINUE/<module>-summary.md`（如继续已有功能）

#### 会话中
- 单次对话不超过安全上限
- 复杂任务自动拆分为子任务

#### 会话结束
1. 追加日志到 `DEVLOG/YYYY-MM-DD.md`
2. 更新 `PROJECT_CONTEXT.md` 的开发进度（如模块完成）

### 4.3 上下文溢出处理

当对话接近安全上限时：
1. **保留优先级：** AGENTS.md > PROJECT_CONTEXT.md > 当前模块 PRD > 相关总结
2. **移除内容：** 历史 DEVLOG、已完成模块的 PRD（可通过 REFERENCES.md 重新引用）
3. **通知用户：** 当上下文即将溢出时，提示用户当前状态

## 5. 大文件处理优化

### 5.1 文档类（doc/PDF）处理流程

```
用户提交大文件
    ↓
TRAE 识别文件类型（doc/PDF）
    ↓
调用本地模型生成 Markdown 摘要（≤500字）
    ↓
将摘要写入 REFERENCES.md
    ↓
后续对话只引用摘要，需要细节时按路径读取原文
```

**摘要内容要求：**
- 文档标题与作者（如有）
- 核心需求/功能描述（2-3句话）
- 关键数据点或约束条件
- 与项目模块的对应关系

### 5.2 Figma 设计稿处理流程

```
用户提供 Figma 链接或 Token
    ↓
TRAE 通过 Figma MCP 获取设计数据
    ↓
按页面/组件拆分导出
    ↓
每个页面生成独立文档（ASSETS/figma/<page>.md）
    ↓
在 REFERENCES.md 中建立索引
```

**页面文档内容要求：**
- 页面名称与描述
- 关键组件列表（按钮、输入框、导航等）
- 交互说明（如有）
- 导出图片（如需可视化参考）

### 5.3 处理参数

| 类型 | 最大摘要长度 | 存储位置 | 引用方式 |
|------|-------------|----------|----------|
| doc/PDF | 500字 | REFERENCES.md | 路径引用 |
| Figma 页面 | 800字 + 图片 | ASSETS/figma/ + REFERENCES.md | 路径+图片引用 |

## 6. 错误处理与恢复

### 6.1 常见错误及处理

| 错误类型 | 处理方式 |
|----------|----------|
| 模型不可用（连接失败） | 重试 3 次 → 记录到 DEVLOG/ → 通知用户 |
| 上下文溢出 | 自动拆分任务 → 记录到 DEVLOG/ → 继续处理 |
| 生成内容异常（乱码、截断） | 重试 1 次 → 如仍失败，记录到 DEVLOG/ → 通知用户 |
| 大文件处理超时 | 降低摘要长度 → 记录到 DEVLOG/ → 通知用户 |

### 6.2 恢复策略
1. 从 `DEVLOG/` 中查找最近的错误记录
2. 根据错误类型选择恢复方式：
   - 连接失败：检查 oMLX 服务状态，确认配置正确
   - 上下文溢出：移除低优先级文档，保留核心文件与当前模块
   - 生成异常：调整模型参数（降低 temperature、减少输出长度）

## 7. 模型切换决策树

### 7.1 决策流程图

```
任务到达
    │
    ▼
┌──────────────────────┐
│ 本地模型是否可用？    │
└──┬───────────────┬───┘
   是              否
    │               ▼
    │         ┌──────────────┐
    │         │ 使用云端模型 │
    │         │ (Claude/GPT) │
    │         └──────────────┘
    ▼
任务复杂度评估：
    │
    ├─ 简单（文本格式化/问答）
    │       → Gemma (26B) — 快速响应，节省资源
    │
    ├─ 中等（代码生成 <500行/技术文档）
    │       → Gemma (26B) — 足够处理中等复杂度
    │
    ├─ 高（代码生成 >500行/长文档分析）
    │       → Qwen (35B) — 需要更大上下文窗口
    │
    └─ 极高（多模块架构设计/跨文件重构）
            → 评估是否切云端模型
              │
              ├─ 有网络且需要更强推理 → Claude/GPT (云端)
              └─ 本地 Qwen 仍可用   → Qwen (35B)
```

### 7.2 任务复杂度评估标准

| 复杂度等级 | 判定条件 | 推荐模型 | 是否可本地完成 |
|-----------|---------|---------|--------------|
| **简单** | 文本格式化、单文件问答、代码片段解释 | Gemma (26B) | ✅ 是 |
| **中等** | 单模块代码生成（<500行）、技术文档编写、简单 UI 实现 | Gemma (26B) | ✅ 是 |
| **高** | 多模块代码生成（500-2000行）、长文档分析、跨文件重构 | Qwen (35B) | ✅ 是（M5 Pro 48GB） |
| **极高** | 全架构设计、多模块联动重构、复杂算法实现 | Qwen (35B) 或云端 | ⚠️ 视情况 |

### 7.3 上下文窗口约束检查

切换模型前，必须检查当前对话的上下文占用：

```
计算当前上下文 Token 数
    │
    ▼
┌─────────────────────────────┐
│ Gemma 上下文窗口使用率？     │
└──┬───────────────┬──────────┘
   ≤60%           >60%
    │               │
    ▼               ▼
Gemma 可继续   Qwen 是否可用？
处理该任务        │
              ┌──┴───────┐
              是          否
               │          ▼
          Qwen 接管     云端模型
          处理           (必须联网)
```

| Gemma 使用率 | Qwen 可用 | 决策 |
|-------------|----------|------|
| ≤60% | — | Gemma 继续处理（无需切换） |
| 60%-80% | ✅ 是 | 切换到 Qwen（Qwen 安全上限为80%） |
| 60%-80% | ❌ 否 | Gemma 继续但限制输出长度 |
| >80% | ✅ 是 | 必须切换到 Qwen 或开启新对话 |
| >80% | ❌ 否 | 开启新对话，保留核心文件，历史通过路径引用 |

### 7.4 网络状态自适应策略

| 网络条件 | 本地模型可用 | 决策 |
|---------|------------|------|
| 有网络 | ✅ 是 | 优先本地模型（快、无隐私风险） |
| 有网络 | ❌ 否 | 云端模型（Claude/GPT），适合极高复杂度任务 |
| 无网络 | ✅ 是 | 本地模型（Gemma/Qwen） |
| 无网络 | ❌ 否 | 记录错误到 DEVLOG/，通知用户等待网络恢复 |

### 7.5 模型切换记录

每次模型切换时，TRAE 应在 `DEVLOG/` 中记录：
- 切换时间
- 原模型 → 目标模型
- 切换原因（上下文溢出/任务复杂度/网络状态）
- 是否影响输出质量

## 10. Token 成本追踪与模型对比

在 AI 辅助开发中，理解每次对话的 Token 消耗和成本构成，有助于优化开发效率和预算分配。本节提供 Token 追踪和多模型对比的完整框架。

### 10.1 Token 估算方法

| 操作类型 | 估算方式 | 示例 |
|----------|---------|------|
| **读取文件** | 按字符数 ÷ 1.3（GPT-4 近似值） | 读取 10KB AGENTS.md ≈ 7,700 tokens |
| **生成回复** | TRAE 输出字符数 ÷ 1.5 | 输出 3KB 代码 ≈ 2,000 tokens |
| **完整对话** | 输入 tokens + 输出 tokens | 读取核心文件(15K) + 代码生成(2K) = 17K tokens |

### 10.2 各模型 Token 成本参考表

| 模型 | 输入价格（每1M tokens） | 输出价格（每1M tokens） | 典型单次对话成本 |
|------|---------------------|---------------------|---------------|
| Gemma (本地) | ¥0（本地运行） | ¥0（本地运行） | 零成本，受限于硬件性能 |
| Qwen (本地) | ¥0（本地运行） | ¥0（本地运行） | 零成本，受限于硬件性能 |
| Claude (云端) | $3.00 / 1M tokens | $15.00 / 1M tokens | ¥3-8/次对话 |
| GPT-4 (云端) | $10.00 / 1M tokens | $30.00 / 1M tokens | ¥8-25/次对话 |
| GPT-4o (云端) | $2.50 / 1M tokens | $10.00 / 1M tokens | ¥2-15/次对话 |

> **注意：** 以上价格为参考值，实际价格以官方定价为准。本地模型（Gemma/Qwen）无 Token 成本，但有硬件折旧和电力成本。

### 10.3 对话 Token 预算分配

单次对话的 Token 应合理分配到不同环节：

```
总 Token 预算（以 Qwen 32K 窗口为例）
    │
    ├─ 永久层加载：~5,000 tokens（AGENTS + PROJECT_CONTEXT + MODEL_CONFIG）
    ├─ 当前模块 PRD：~3,000 tokens（按需加载）
    ├─ 相关总结文件：~2,000 tokens（按需加载）
    ├─ 对话交互（输入）：~15,000 tokens（用户问题 + 上下文）
    └─ 对话交互（输出）：~7,000 tokens（代码、文档、分析结果）
```

**安全原则：** 单次对话总 Token 不超过模型窗口的 80%，预留 20% 缓冲空间。

### 10.4 Token 效率优化策略

| 策略 | 节省比例 | 说明 |
|------|---------|------|
| **使用本地模型处理简单任务** | 100%（零成本） | 文本格式化、问答等轻量任务优先本地模型 |
| **精简文件读取** | 30-50% | 只读取需要的文件，不加载全部历史文档 |
| **决策日志替代重复讨论** | 40-60% | 通过决策日志减少3轮内的恢复时间 |
| **CONTINUE/总结复用** | 20-40% | 新对话通过总结快速恢复，减少上下文重建轮数 |
| **模块化开发（≤500行）** | 30-50% | 小任务单次对话 Token 消耗更低 |

### 10.5 多模型对比矩阵

| 维度 | Gemma (26B) | Qwen (35B) | Claude (云端) | GPT-4o (云端) |
|------|------------|-----------|--------------|--------------|
| **Token 成本** | ¥0（本地） | ¥0（本地） | $15/M输出 | $30/M输出 |
| **响应速度** | 快（秒级） | 中（数秒-数十秒） | 慢（数十秒） | 中（数秒-数十秒） |
| **代码质量** | 良好（<500行） | 优秀（>500行） | 优秀 | 优秀 |
| **上下文窗口** | ~8K tokens | ~32K tokens | 200K tokens | 128K tokens |
| **离线可用** | ✅ | ✅ | ❌ | ❌ |
| **隐私保护** | 完全本地 | 完全本地 | 数据传输云端 | 数据传输云端 |
| **适用场景** | 日常开发、小任务 | 复杂功能、长文档 | 架构设计、跨模块 | 全场景备用 |

### 10.7 成本控制建议

1. **优先本地模型：** 所有可以在 Gemma/Qwen 完成的任务，不切换到云端模型
2. **批量处理：** 将多个相关小任务合并到单次对话，减少重复的永久层加载成本
3. **利用决策日志：** 通过决策日志减少新对话的恢复时间（3轮内恢复到理解水平）
4. **监控异常消耗：** 单次对话 Token 超过平时平均值的 3 倍时，检查是否有异常（如死循环、过度输出）
5. **云端模型作为最后手段：** 仅在本地模型无法处理且任务复杂度为"极高"时，才使用云端模型
