# S06 — Agent Task Granularity

> 对应 ChatGPT 方案 §7。定义 AI Agent 的任务拆分粒度标准，避免"一次做太多"导致的输出质量下降。

---

## 1. 核心原则

### ❌ 不要做的事

```
❌ "开发整个首页"
   → 范围太大，包含 UI + 网络 + 数据 + 动画 + 可访问性

❌ "重构整个 AuthService"
   → 涉及太多依赖，风险高

❌ "写所有测试用例"
   → 无优先级，质量不可控
```

### ✅ 应该做的事

```
✅ 将大任务按层级拆分：
   分析 → 建模 → UI → 网络 → 动画 → 可访问性 → 测试 → 审查

✅ 每个子任务：
   - 单一职责（只做一件事）
   - 有明确输入和输出
   - 可在单次对话内完成（≤500 行代码，≤20 个文件）
   - 有独立可验证的验收标准
```

---

## 2. 任务粒度分级标准

| 粒度等级 | 代码规模 | 涉及文件 | 单次对话 | 示例 |
|---------|---------|---------|---------|------|
| **原子级** | <50 行 | 1 个 | ✅ 适合 | 实现一个 Toggle 组件、修复一个拼写错误 |
| **组件级** | 50-200 行 | 1-3 个 | ✅ 适合 | 实现 LoginView、编写 ViewModel 单元测试 |
| **模块级** | 200-500 行 | 3-10 个 | ⚠️ 需谨慎 | 实现完整登录模块（View + ViewModel + Service） |
| **子系统级** | 500-2000 行 | 10-20 个 | ❌ 应拆分 | 实现认证系统（登录 + 注册 + 密码重置 + Token管理） |
| **架构级** | >2000 行 | >20 个 | ❌ 必须拆分 | 全应用架构重构 |

## 3. 任务拆分决策树

```
收到用户任务
    │
    ▼
【判断1】涉及几个文件？
    ├── 1-3 个 → 进入判断2
    ├── 3-10 个 → 评估是否可拆分为多个独立子任务
    └── >10 个 → 必须先拆分，再逐个执行
         │
         ▼
【判断2】预估代码行数？
    ├── <200 行 → 可直接执行（单次对话）
    ├── 200-500 行 → 可执行，但需在对话中设置检查点
    └── >500 行 → 必须拆分为多个子任务
         │
         ▼
【判断3】是否涉及多个独立关注点？
    ├── 单一关注点 → 直接执行
    └── 多个关注点（如 UI + 网络 + 数据） → 拆分为独立子任务
```

## 4. Apple App 完整分解示例

### 以"登录功能"为例

```
用户请求："开发登录功能"
          │
          ▼
【分析阶段】不写代码，只分析需求
  任务 1：读取 PRD/login.md，输出功能点清单
  任务 2：识别依赖模块（网络层、存储层、UI 组件库）

【建模阶段】设计数据结构
  任务 3：设计 LoginModel（email, password, authToken）
  任务 4：设计 AuthState（idle, loading, success, error）

【UI 层】实现界面
  任务 5：实现 LoginView.swift（邮箱输入、密码输入、登录按钮）
  任务 6：实现登录按钮的加载和错误状态 UI

【网络层】实现后端交互
  任务 7：实现 AuthService.login(email:password:) 网络请求
  任务 8：实现 Token 管理和自动刷新

【动画层】提升体验
  任务 9：实现登录按钮的加载动画
  任务 10：实现登录成功/失败的过渡动画

【可访问性层】确保无障碍
  任务 11：添加 VoiceOver 标签和提示
  任务 12：适配动态字体大小

【测试层】质量保证
  任务 13：编写 AuthViewModel 单元测试
  任务 14：编写 LoginView UI 测试

【审查层】最终检查
  任务 15：Code Review（检查错误处理、内存泄漏、安全漏洞）
  任务 16：生成 CONTINUE/login-summary.md
```

### 关键拆分原则

```
每一层完成后再进入下一层：
  ✅ 分析 → 建模 → UI → 网络 → 动画 → 可访问性 → 测试 → 审查

不要跨层混合：
  ❌ UI + 网络 + 数据 在一个子任务中
  ❌ 测试 + 审查 在开发完成前就执行

每个子任务结束后的检查点：
  □ 验收标准是否全部满足？
  □ 是否需要在 DEVLOG/ 中记录？
  □ 是否需要生成决策日志？
  □ 是否触发下一子任务的条件已满足？
```

## 5. 不同项目类型的粒度参考

### Apple App (SwiftUI)

```
页面级功能分解：
  PRD 分析 → 数据模型 → View → ViewModel → Service → 动画 → 可访问性 → 测试 → 总结
```

### Web 应用 (React / Next.js)

```
路由级功能分解：
  PRD 分析 → 类型定义 → Component → Hook/Store → API Route → 样式 → 响应式 → 测试 → 总结
```

### 后台服务

```
API 端点级分解：
  PRD 分析 → Schema 定义 → Controller → Service → Repository → 中间件 → 错误处理 → 测试 → 总结
```

### 游戏 (Unity / Godot)

```
玩法系统级分解：
  设计文档 → 数据结构 → 核心逻辑 → 输入处理 → 视觉反馈 → 音效 → 性能优化 → 测试 → 总结
```

## 6. 任务执行顺序依赖

### 依赖图示例（登录模块）

```
PRD 分析（任务 1）
    │
    ├──→ 依赖识别（任务 2）
    │       │
    │       ├──→ 数据模型（任务 3）
    │       │       │
    │       │       └──→ 状态机（任务 4）
    │       │               │
    │       └──→ 网络层（任务 7）←──┐
    │               │               │
    ├──→ UI 层（任务 5）           │
    │       │                      │
    │       ├──→ 状态 UI（任务 6）  │
    │       │                      │
    │       ├──→ 动画（任务 9）    │
    │       │                      │
    │       └──→ 可访问性（任务 11）│
    │               │               │
    └──→ 测试（任务 13）────────────┘
            │
            └──→ 审查（任务 15）
                    │
                    └──→ 总结（任务 16）
```

**并行机会**：
- 数据模型（任务 3-4）与网络层（任务 7-8）可并行
- UI 层（任务 5-6）可在数据模型完成后与网络层并行
- 动画（任务 9-10）和可访问性（任务 11-12）可并行

**串行依赖**：
- PRD 分析必须在所有任务之前
- 测试必须在对应模块完成后
- 审查必须在所有开发完成后

## 7. 任务拆分质量检查清单

在开始执行拆分后的每个子任务前：

```
□ 子任务是否只做一件事？       （单一职责）
□ 是否能在单次对话内完成？     （≤500 行代码，≤20 文件）
□ 输入是否明确？               （引用了具体文件和章节）
□ 验收标准是否可验证？         （有明确的"完成"定义）
□ 依赖是否已满足？             （前置任务已完成）
□ 是否与其他子任务有冲突？     （不修改同一文件的同一区域）
□ 是否有明确的输出产物？       （代码文件/文档/测试结果）
```

## 8. 常见反模式

### ❌ 超大任务

> "实现完整的用户系统，包括注册、登录、密码重置、个人资料、头像上传"

**应拆分为**（至少 5 个子任务）：
1. 注册模块
2. 登录模块
3. 密码重置模块
4. 个人资料编辑
5. 头像上传

### ❌ 跨层混合

> "做一个登录页面，包括 UI、网络请求、数据库存储"

**应拆分为**：
1. ONLY UI：LoginView.swift（不含网络和存储逻辑）
2. ONLY 网络：AuthService.swift
3. ONLY 存储：TokenStorage.swift

### ❌ 无优先级拆分

> "写所有测试"

**应按优先级拆分**：
1. 核心业务逻辑的单元测试（高优先级）
2. UI 交互测试（中优先级）
3. 边界情况和性能测试（低优先级）
