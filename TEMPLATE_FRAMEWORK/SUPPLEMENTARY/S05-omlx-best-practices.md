# S05 — oMLX Best Practice

> 对应 ChatGPT 方案 §14。针对 Apple Silicon 上 oMLX 托管本地模型（Gemma/Qwen）的性能优化与运维指南。

---

## 1. oMLX 概述

oMLX 是 Apple Silicon 上的本地 LLM 推理引擎，利用 Metal GPU 加速和统一内存架构（UMA），在 M 系列芯片上高效运行量化模型。

### 1.1 推荐硬件配置

| 芯片 | 统一内存 | 推荐模型 | 最大并发 |
|------|---------|---------|---------|
| M1 Pro | 16GB | Gemma 9B (Q4) | 1 |
| M2 Pro | 32GB | Qwen 14B (Q4) + Gemma 9B (Q4) | 2 |
| M3 Max | 36GB | Qwen 32B (Q4) | 2 |
| M4 Max | 48GB+ | Qwen 35B (Q4) + Gemma 27B (Q4) | 2-3 |

### 1.2 当前项目配置（参考）

```
MacBook Pro M5 Pro
48GB Unified Memory

oMLX 实例：
├── Gemma 4 27B (Q4_K_M) — 日常开发主力
└── Qwen 3.6 35B (Q4_K_M) — 复杂任务/长文档
```

## 2. Context 窗口管理

### 2.1 窗口大小与内存关系

| 上下文窗口 | 额外内存占用（Q4） | 推荐场景 |
|-----------|-----------------|---------|
| 4K tokens | ~0.5 GB | 简单问答、代码片段 |
| 8K tokens | ~1 GB | 单文件开发、小 PRD |
| 16K tokens | ~2 GB | 模块级开发 |
| 32K tokens | ~4 GB | 长文档分析、跨文件重构 |

**安全公式**：模型内存 + 上下文内存 + 系统预留 8GB ≤ 总统一内存

### 2.2 上下文溢出预防

```
检测策略：
  每 10 轮对话检查一次当前 Token 使用率
    │
    ├── <60% → 正常，继续
    ├── 60%-80% → 关注，限制新文件加载
    └── >80% → 触发溢出处理：
         ├── 1. 移除临时层文件（历史 DevLog、中间产物）
         ├── 2. 压缩稳定层引用（路径代替全文）
         └── 3. 如仍超限 → 切换到大上下文模型或开启新对话
```

### 2.3 上下文清理时机

| 触发条件 | 行动 |
|---------|------|
| 切换开发模块 | 卸载旧模块 PRD，加载新模块 PRD |
| 模块完成 | 仅保留 CONTINUE/ 总结，卸载详细 PRD |
| 新对话开始 | 从零构建，仅加载永久层 |
| 单次对话 >25 轮 | 强制生成中间总结，建议开启新对话 |

## 3. Memory（统一内存）管理

### 3.1 内存分配优先级

```
48GB 统一内存分配策略：
├── 系统 + 其他应用：8GB（固定预留）
├── oMLX 模型权重：~16GB（Gemma 27B Q4）
├── oMLX 模型权重：~20GB（Qwen 35B Q4，按需加载）
├── KV Cache / 上下文：~2-4GB
└── 可用缓冲：0-2GB
```

### 3.2 内存压力监控

| 内存压力 | macOS 颜色 | 症状 | 行动 |
|---------|-----------|------|------|
| 绿色 | 绿色 | 正常 | 无需操作 |
| 黄色 | 黄色 | 响应略慢 | 检查是否有僵尸进程 |
| 红色 | 红色 | 模型可能 OOM | 立即卸载非活跃模型，关闭不需要的应用 |
| 压缩 | — | Swap 增加 | 降低上下文窗口大小，减少并发模型数 |

### 3.3 多模型共存策略

```
场景 A：仅需 Gemma（日常开发）
  → 仅加载 Gemma，释放 Qwen 所占内存给系统

场景 B：Gemma + Qwen 共存（复杂任务需要切换）
  → 两个模型同时加载，但仅激活一个
  → 内存占用：16GB + 20GB = 36GB，48GB 安全

场景 C：两个模型都活跃（极少见）
  → 总内存接近上限，降低上下文窗口为 8K
  → 如仍不够 → 关闭一个模型
```

## 4. SSD Cache（磁盘缓存）

### 4.1 缓存策略

oMLX 使用 SSD 缓存来加速模型加载和上下文切换：

```bash
# 缓存位置
~/.cache/omlx/

# 检查缓存大小
du -sh ~/.cache/omlx/

# 清理过期缓存（>30天未使用）
find ~/.cache/omlx/ -atime +30 -delete
```

### 4.2 缓存预热

在项目初始化或长期中断后恢复时：

```
预热步骤：
1. 加载 Gemma 模型（首次加载 ~30 秒，缓存后 ~5 秒）
2. 发送一次简单请求（"Hello"）激活推理引擎
3. 确认响应时间 <100ms
4. 如需要，预热 Qwen（首次加载 ~45 秒）
```

### 4.3 缓存失效处理

| 场景 | 处理 |
|------|------|
| oMLX 版本升级 | 清除旧缓存，重新预热 |
| 模型更新（新 GGUF） | 删除旧模型缓存文件 |
| 磁盘空间不足 | 清理 >30 天未使用的缓存 |
| 模型加载变慢 | 检查 SSD 健康度，必要时清理重建 |

## 5. Profile（性能剖析）

### 5.1 关键性能指标

| 指标 | Gemma 目标 | Qwen 目标 | 测量方式 |
|------|-----------|----------|---------|
| 首 Token 延迟 | <100ms | <200ms | oMLX 内置计时 |
| 生成速度 | >30 tok/s | >20 tok/s | 计算输出 tokens / 耗时 |
| 上下文加载 | <1s | <2s | 从请求发送到首 Token |
| 模型切换 | <5s | — | 从卸载旧模型到新模型就绪 |

### 5.2 性能劣化诊断

| 症状 | 可能原因 | 排查步骤 |
|------|---------|---------|
| 生成速度骤降 | 温度过高导致 Swift 降频 | 检查电池状态，切换到电源供电 |
| 首 Token 延迟 >1s | 上下文过大 | 检查 Token 使用率，清理上下文 |
| 偶发超时 | 其他应用抢占 GPU | Activity Monitor 检查 GPU 占用 |
| 模型加载超时 | SSD 缓存损坏 | 清除缓存后重试 |

### 5.3 性能日志格式

每次模型切换或性能异常时，记录到 DEVLOG/：

```markdown
## oMLX 性能记录 - YYYY-MM-DD
- **模型：** Gemma / Qwen
- **任务类型：** 代码生成 / 文档分析 / 对话
- **输入 Tokens：** [数字]
- **输出 Tokens：** [数字]
- **首 Token 延迟：** [ms]
- **生成速度：** [tok/s]
- **总耗时：** [秒]
- **异常：** [如有，描述症状]
```

## 7. Runtime（运行时管理）

### 7.1 启动检查清单

```
每次启动 oMLX 后：
□ 确认模型已加载（oMLX status 或 API /v1/models）
□ 发送健康检查请求（"ping" → 预期响应 "pong" 或正常回复）
□ 检查可用内存（macOS Activity Monitor）
□ 确认上下文窗口大小与配置一致
□ 如有异常，记录到 DEVLOG/
```

### 7.2 运行时监控

| 监控项 | 频率 | 记录位置 |
|--------|------|---------|
| 内存使用 | 每次对话开始时 | — |
| 响应延迟 | 每次模型调用后 | DEVLOG/ |
| 错误率 | 每周汇总 | DEVLOG/weekly-* |
| Token 消耗 | 每周汇总 | DEVLOG/weekly-* |

### 7.3 优雅关闭

```
关闭 oMLX 前：
1. 卸载所有模型（释放统一内存）
2. 确认无待处理请求
3. 记录本次运行时长和 Token 消耗到 DEVLOG/
4. 正常退出 oMLX 进程（不要 force kill）
```

## 8. Memory Guard（内存保护）

### 8.1 自动保护机制

```
内存使用率监控（每次推理前检查）：
    │
    ├── <70% → 正常
    ├── 70%-85% → 警告：降低新内容加载
    └── >85% → 危险：
         ├── 1. 暂停加载新模型
         ├── 2. 清理临时层（历史 DevLog）
         ├── 3. 卸载非活跃模型（如只保留 Gemma）
         └── 4. 如仍超限 → 通知用户，拒绝新推理请求
```

### 8.2 Swap 防护

| Swap 使用量 | 行动 |
|------------|------|
| 0 GB | 正常 |
| 0-2 GB | 关注，减少并发模型 |
| 2-5 GB | 警告，卸载非活跃模型 |
| >5 GB | 严重，立即卸载所有模型，仅保留活动模型 |

### 8.3 内存泄漏排查

| 症状 | 排查方向 |
|------|---------|
| 内存持续增长 | 检查是否有未释放的上下文缓存 |
| 关闭模型后内存未释放 | 可能是 oMLX bug，重启 oMLX 服务 |
| Swap 持续增加 | 加载了过大模型，降级量化等级或使用更小模型 |
