# S04 — Long-term Project Strategy

> 对应 ChatGPT 方案 §16。针对持续3个月、半年、1年的AI辅助开发项目的长期策略，包含AI如何持续理解项目、跨月恢复协议深度版、知识衰减曲线管理。

---

## 1. 长期项目挑战与应对策略

### 1.1 核心问题：AI记忆衰减

```
时间（月）    AI对项目理解的准确度（无补偿机制）
0-1           ████████████ 高 — 有完整PRD + DevLog
1-2           ██████████   中高 — 月度摘要补偿部分衰减
2-3           ████████     中 — 需要季度审计恢复上下文
3+            █████        低 — 必须依赖外部记忆机制
```

**核心原则：通过完善的外部记忆（文件系统文档），使AI在项目任何阶段都能恢复到接近初始的理解水平。**

### 1.2 长期项目的特殊挑战

| 挑战 | 3个月项目 | 6个月项目 | 1年+项目 |
|------|---------|---------|---------|
| **模块数量** | 3-8个模块 | 8-20个模块 | 20+模块，可能分阶段发布 |
| **API变更风险** | 低（iOS版本未大变） | 中（可能跨一个大版本） | 高（多次API变更） |
| **团队变化** | 通常不变 | 可能有人力调整 | 高概率有人员变更 |
| **需求变更频率** | 低-中 | 中 | 高（市场/竞争驱动） |
| **AI恢复成本** | 低（3-5轮对话） | 中（5-10轮对话） | 高（10+轮对话，除非有完善文档） |

## 2. 3个月项目策略

### 2.1 阶段划分

```
第1个月：核心功能开发（MVP）
├── 完成登录、首页等核心模块
├── 建立基础架构（MVVM/Clean Architecture）
└── 生成首批月度摘要

第2个月：功能完善
├── 完成次要功能模块（设置、通知等）
├── 开始性能优化
└── 建立决策日志体系

第3个月：测试与发布准备
├── 全面测试（单元/集成/UI自动化）
├── App Store提交准备
└── 生成最终季度审计报告
```

### 2.2 维护频率要求

| 维护类型 | 频率 | 3个月项目特殊要求 |
|---------|------|-----------------|
| **每日清单**（§15） | 每次对话 | 必须执行，确保每个会话都有明确的开始和结束 |
| **周维护**（§12.1） | 每7天 | 3个月项目至少执行4次周维护，记录到PROJECT_CONTEXT.md |
| **月维护**（§12.2） | 每30天 | 至少执行3次月维护，每次生成ARCHIVE/monthly-YYYY-MM.md |
| **季度审计**（§12.3） | 每90天 | 项目结束时执行，生成完整的季度审计报告 |

### 2.3 关键里程碑文档

| 里程碑 | 必须生成的文档 | 存储位置 |
|--------|--------------|---------|
| **项目启动**（第1天） | AGENTS.md, PROJECT_CONTEXT.md, MODEL_CONFIG.md | TEMPLATE_FRAMEWORK/ |
| **核心架构确定**（第7-14天） | 架构决策日志 + 目录结构最终版 | CONTINUE/decision-log/ |
| **核心功能完成**（第30天） | 首个ARCHIVE/monthly-YYYY-MM.md | ARCHIVE/ |
| **次要功能完成**（第60天） | 第二个ARCHIVE/monthly-YYYY-MM.md + 决策日志汇总 | ARCHIVE/ + CONTINUE/decision-log/ |
| **项目发布**（第90天） | 完整的季度审计报告 + 最终项目健康报告 | ARCHIVE/quarterly-YYYY-Q3.md |

## 3. 半年项目策略（含中间发布）

### 3.1 阶段划分

```
第1-2个月：MVP开发（V1.0）
├── 核心功能模块开发
└── 内部测试版发布

第3-4个月：V1.1功能迭代
├── 基于用户反馈的功能改进
├── 新增次要模块（如有）
└── V1.1发布

第5-6个月：V2.0规划与开发
├── 架构评估（是否需要重构？）
├── V2.0新功能开发
└── 发布准备或进入下一周期
```

### 3.2 中期策略调整机制

在项目的第2个月末（约60天），TRAE应自动执行策略调整检查：

```
第2个月末策略检查
    │
    ▼
1. 评估当前开发进度与计划的偏差
    ├── 提前或按计划 → 继续执行原计划
    └── 落后于计划 → 评估是否需要调整scope（full→small）

2. 检查技术栈是否仍然适用
    ├── iOS是否有新版本发布？
    ├── SwiftUI是否有重大API变更？
    └── 本地模型是否有升级？

3. 评估团队/项目范围变化
    ├── 是否有人力增减？
    ├── 是否有新的功能需求加入？
    └── 发布时间表是否变更？

4. 输出《中期策略调整建议》
    └── 存储到：ARCHIVE/interim-review-YYYY-MM.md
```

### 3.3 半年项目文档密度要求

| 文档类型 | 3个月项目频率 | 6个月项目频率（翻倍） |
|---------|-------------|-------------------|
| DevLog | 每天1条 | 每天1条（相同） |
| 周总结 | 每周1份 | 每周1份（相同） |
| 月摘要 | 每月1份 | 每月1份（相同，但内容更丰富） |
| 决策日志 | 重要决策时 | 重要决策时（相同，但数量更多） |
| 策略调整检查 | 不适用 | 每2个月1次（新增） |
| 季度审计 | 项目结束时1次 | 每3个月1次（共2次） |

## 4. 1年+项目策略（多版本迭代）

### 4.1 年度项目架构

```
Year 1: V1.x — 核心功能 + 市场验证
├── Q1: MVP开发 → V1.0发布
├── Q2: 用户反馈迭代 → V1.1/V1.2发布
├── Q3: 功能扩展 → V1.5发布
└── Q4: 架构评估 + V2.0规划

Year 2+: V2.x — 规模化 + 新功能线
├── 基于V1.x的用户数据和反馈
├── 可能的技术栈升级（iOS新版本、Swift新版本）
└── 可能的新功能线或平台扩展（watchOS/iPadOS等）
```

### 4.2 年度知识管理体系

对于1年+的项目，必须建立分层知识管理体系：

```
知识管理层次结构（长期项目专用）
│
├── L1: 永久层（始终加载，不随时间变化）
│   ├── AGENTS.md — 项目规则（仅在项目方向重大变更时更新）
│   ├── PROJECT_CONTEXT.md — 项目核心信息（每季度更新概述部分）
│   └── MODEL_CONFIG.md — 模型配置（每月审查）
│
├── L2: 稳定层（随项目演进，但保持向后兼容）
│   ├── PRD/ — 所有历史功能需求文档（保留全部，不删除）
│   ├── CONTINUE/ — 所有模块总结（保留全部）
│   └── ARCHIVE/ — 月度摘要 + 季度审计（保留全部）
│
├── L3: 索引层（帮助AI快速定位历史知识）
│   ├── REFERENCES.md — 大文件索引（持续更新）
│   └── PROJECT_CONTEXT.md → "相关文档索引"章节（定期整理）
│
└── L4: 衰减层（随时间降低优先级，但不立即删除）
    ├── DevLog/ — 超过90天的日志（标记为可弃，季度审计时处理）
    └── 已完成且无后续引用的中间文件（同上处理）
```

### 4.3 年度项目维护日历

| 月份 | 特殊维护任务 |
|------|-------------|
| **每月** | 标准月维护（§12.2），生成ARCHIVE/monthly-YYYY-MM.md |
| **每季度** | 标准季审计（§12.3）+ iOS版本变更检查 + 模型升级检查 |
| **每6个月** | 中期策略调整（§3.2）+ 架构健康评估 |
| **每年** | 年度全面审计：scope重新评估、技术栈全面升级检查、项目方向确认 |

## 5. 跨月恢复协议（深度版）

### 5.1 跨月恢复决策树

```
距离上次开发 >7天，开启新对话
    │
    ▼
【检查1】项目运行总时长？
    ├── <30天 → 标准恢复流程（PROJECT_CONTEXT.md §9.6）
    ├── 30-90天 → 加载最近1个月ARCHIVE/月度摘要
    └── >90天 → 加载最近3个月ARCHIVE/月度摘要 + 最近季度审计报告
         │
         ▼
    【检查2】最后完成的模块是否有CONTINUE/总结？
        ├── 有 → 加载该总结，恢复到开发上下文
        └── 无 → 手动阅读对应PRD + DevLog重建理解

【检查3】是否有未解决的已知问题？
    ├── 有 → 在恢复后优先处理或重新评估
    └── 无 → 正常开始新开发任务

输出：向用户展示"恢复后的项目状态摘要"
```

### 5.2 跨月恢复检查清单

开启新对话前，TRAE必须执行：

```
□ 读取永久层文件（AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md）
□ 检查项目运行总时长（决定加载多少ARCHIVE/内容）
□ 读取最近[1-3]个月的ARCHIVE/月度摘要
□ 列出所有已完成模块及其总结（来自PROJECT_CONTEXT.md和CONTINUE/）
□ 检查是否有标记为"可弃"的文件需要处理
□ 向用户展示"上次开发时项目状态"：
   ├── 最后完成的模块名称和日期
   ├── 当前正在开发的模块（如有）
   ├── 已知问题和待办事项
   └── 自上次开发以来的重要变更（iOS更新、模型升级等）
□ 确认用户理解当前状态后，才开始新开发任务
```

## 6. 知识衰减曲线管理与补偿机制

### 6.1 不同文档类型的衰减速度

| 文档类型 | 衰减速度 | 有效期限 | 补偿方式 |
|---------|---------|---------|---------|
| **AGENTS.md** | 极慢（几乎不衰减） | 项目全周期 | 仅在规则变更时更新 |
| **PROJECT_CONTEXT.md** | 慢（概述部分） | 3-6个月 | 季度审计时更新概述 |
| **PRD（已完成模块）** | 中（需求可能变更） | 1-3个月 | 月维护时确认PRD仍有效 |
| **CONTINUE/总结** | 中（可能遗漏新发现） | 1-3个月 | 模块相关决策更新时补充 |
| **DevLog** | 快（细节遗忘） | <30天 | 月/季度摘要中提炼关键信息 |
| **决策日志** | 慢（决策理由不变） | 3-6个月 | 季度审计时回顾有效性 |

### 6.2 衰减预警与补偿行动表

| 文档类型 | 预警条件（超过多久未更新） | 补偿行动 |
|---------|------------------------|---------|
| PROJECT_CONTEXT.md | >7天未更新模块状态 | 提醒用户确认项目状态 |
| ARCHIVE/月度摘要 | 项目>30天但无月度文件 | 提醒执行首次月维护 |
| DevLog | >14天空白期 | 提醒检查项目是否中断 |
| 决策日志 | >30天未回顾的决策 | 月维护时扫描并标记审查 |
| MODEL_CONFIG.md | >30天未审查 | 提醒检查本地模型版本 |

## 7. 长期项目的scope调整策略

### 7.1 Scope动态调整决策树

```
项目运行超过[30/60/90]天
    │
    ▼
【检查】当前scope是否仍然合理？
    │
    ├── 模块数量增长 → 
    │   ├── <8个模块 → scope保持"full"
    │   ├── 8-15个模块 → 评估是否拆分为"full"（核心）+"small"（次要）
    │   └── >15个模块 → 考虑拆分项目或使用"full" + Worktree策略
    │
    ├── 开发进度落后 → 
    │   ├── 落后<20% → 继续原计划，加强周维护频率
    │   ├── 落后20-50% → 评估是否调整scope（full→small）
    │   └── 落后>50% → 必须重新评估scope，可能降级为"mvp"
    │
    └── 需求发生重大变化 → 
        ├── 新增核心功能 → scope保持或升级为"full"（如之前是"small"）
        └── 功能被取消 → scope可能降级（full→small或small→mvp）
```

### 7.2 Scope调整影响评估表

| 原scope | 调整后scope | 影响 | TRAE行动 |
|---------|-----------|------|---------|
| full → small | 跳过TESTING/、DEPLOYMENT/ | 更新PROJECT_CONTEXT.md的scope字段，通知用户跳过的模块 |
| full → mvp | 仅保留核心PRD模块 | 归档非核心模块到ARCHIVE/，更新所有相关占位符状态 |
| small → full | 恢复TESTING/、DEPLOYMENT/ | 生成对应模块的占位符文件，更新PROJECT_CONTEXT.md |
| small → mvp | 进一步精简 | 同上 |

## 8. 长期项目健康度仪表盘（年度版）

### 8.1 年度项目关键指标

| 指标 | 优秀 | 良好 | 警告 | 危险 |
|------|-----|------|-----|------|
| **模块完成度** | >70% | 40-70% | 20-40% | <20% |
| **月维护完成率** | 100% | ≥80% | 50-80% | <50% |
| **决策日志密度** | >3个/月 | 2-3个/月 | 1-2个/月 | <1个/月 |
| **记忆衰减预警数** | 0-1个 | 2-3个 | 4-5个 | >5个 |
| **僵尸模块数** | 0个 | ≤1个 | 2-3个 | >3个 |

### 8.2 年度项目自动报告生成

每季度结束时，TRAE应自动生成《季度健康报告》，包含：

```markdown
## 季度健康报告 — [项目名称] Q[X] [年份]

### A. 项目整体指标
- 总模块数：[X]个，已完成：[Y]个（[Z]%）
- 月维护完成率：[A/B]次（[C]%）
- 决策日志总数：[D]个

### B. 本月重点完成
1. [模块名] — 完成日期：YYYY-MM-DD
2. ...

### C. 风险预警汇总
- 🟡 [预警1] — 状态：[已处理/处理中/未开始]
- 🟠 [预警2] — 状态：...

### D. 下季度计划
1. [计划模块/任务]
2. ...

### E. 策略调整建议（如适用）
- [基于本季度表现提出的scope或计划调整建议]
```
