# S03 — Model Switching Decision Tree

> 对应 ChatGPT 方案 §13。完整的模型切换决策树，包含上下文窗口约束检查、网络状态自适应、多模型对比矩阵。

---

## 1. 模型切换决策树（总览）

```
任务到达
    │
    ▼
【判断1】任务复杂度评估
    ├── 简单（<50行代码/单文件） → 本地模型：Gemma
    ├── 中等（50-200行/单模块） → 本地模型：Qwen
    └── 复杂（>200行/跨模块/架构级） → 进入判断2
         │
         ▼
    【判断2】是否有网络？
         ├── 有网络 → 进入判断3（云端模型）
         └── 无网络 → 本地模型：Qwen（最大上下文模式）
              │
              ▼
         【判断3】是否需要创造性推理？
              ├── 是 → 云端模型：Claude/GPT
              └── 否 → 本地模型：Qwen
```

## 2. 各模型能力画像与适用场景

### 2.1 Gemma（本地，oMLX）

**能力特征：**
- 上下文窗口：4K-8K tokens（较小）
- 推理速度：快（本地运行，<100ms响应）
- 代码生成质量：中等（简单到中等复杂度）
- 知识范围：通用编程 + 基础Apple生态

**适用场景（✅ 推荐）：**
- [x] 简单UI组件编写（<30行SwiftUI代码）
- [x] 变量命名建议
- [x] 简单错误修复（单文件内）
- [x] 代码注释生成
- [x] 小型工具函数实现

**不适用场景（❌ 避免）：**
- [ ] 跨模块架构设计
- [ ] 复杂算法实现（>100行）
- [ ] 需要大量外部知识的问题
- [ ] 多文件协调修改

**Prompt示例：**
> "用SwiftUI写一个20行的Toggle组件，实现深色模式切换功能。文件路径：Views/ThemeToggle.swift。"

---

### 2.2 Qwen（本地，oMLX）

**能力特征：**
- 上下文窗口：32K-128K tokens（中等到较大）
- 推理速度：中等（本地运行，100-500ms响应）
- 代码生成质量：中高（可处理中等复杂度任务）
- 知识范围：通用编程 + Apple生态 + 部分领域知识

**适用场景（✅ 推荐）：**
- [x] 中等复杂度模块开发（50-200行代码）
- [x] 单文件重构（<10个函数）
- [x] 数据模型设计
- [x] 简单网络层封装
- [x] 单元测试编写

**不适用场景（❌ 避免）：**
- [ ] 跨模块架构设计（>3个模块交互）
- [ ] 复杂动画系统实现
- [ ] 需要最新API知识的问题（>3个月前的API变更）

**Prompt示例：**
> "实现AuthViewModel.swift，包含邮箱密码登录、Face ID登录、Token刷新三个功能。参考PRD/login.md的需求描述，使用MVVM架构。输出完整的ViewModel代码和对应的测试用例。"

---

### 2.3 Claude（云端）

**能力特征：**
- 上下文窗口：200K+ tokens（极大）
- 推理速度：慢（网络延迟，500ms-2s响应）
- 代码生成质量：高（复杂架构和创造性推理）
- 知识范围：全领域 + 最新API

**适用场景（✅ 推荐）：**
- [x] 跨模块架构设计
- [x] 复杂系统重构（>5个文件协调修改）
- [x ] 创造性UI设计（需要大量权衡判断）
- [x] 性能优化方案评估
- [x] 安全审计和渗透测试建议

**不适用场景（❌ 避免）：**
- [ ] 简单UI组件（浪费云端资源）
- [ ] 无网络环境（无法使用）
- [ ] 高频交互场景（响应延迟影响体验）

**Prompt示例：**
> "当前项目有登录、首页、设置三个模块。请设计一个统一的数据同步架构，要求：1) 支持离线优先；2) 模块间数据一致性保证；3) 内存使用控制在50MB以内。输出架构图、关键接口定义和已知风险。"

---

### 2.4 GPT（云端）

**能力特征：**
- 上下文窗口：128K+ tokens（大）
- 推理速度：中等（网络延迟，300ms-1.5s响应）
- 代码生成质量：高（擅长具体实现细节）
- 知识范围：全领域 + 最新API

**适用场景（✅ 推荐）：**
- [x] 具体代码实现细节优化
- [x ] 最新API适配（GPT的知识更新通常更快）
- [x] 多语言/国际化实现
- [x] Accessibility（辅助功能）合规检查

**不适用场景（❌ 避免）：**
- [ ] 简单UI组件（本地模型足够）
- [ ] 无网络环境

**Prompt示例：**
> "当前SettingsView.swift的Accessibility评分为C级（通过VoiceOver测试时有以下问题：1) 列表项缺少语义标签；2) 动态类型未适配。请提供具体的修改方案，使评分提升到A级。参考Apple Accessibility Guidelines。"

## 3. 上下文窗口约束检查

### 3.1 上下文窗口使用决策矩阵

| 任务涉及文件数 | 总代码行数 | 推荐模型 | 理由 |
|---------------|-----------|---------|------|
| 1个文件，<50行 | <50行 | Gemma | 完全在Gemma能力范围内 |
| 1个文件，50-200行 | <500行 | Qwen | 在Qwen上下文窗口内，本地运行 |
| 1个文件，>200行 | >500行 | Qwen（拆分）或 Claude | 考虑上下文窗口限制，可能需要拆分任务 |
| 2-3个文件 | <1000行 | Qwen | 在Qwen上下文窗口内 |
| 3-5个文件 | 1000-5000行 | Claude/GPT | 需要云端模型的大上下文窗口 |
| >5个文件或>5000行 | 全项目范围 | Claude + 拆分策略 | 使用Claude进行架构设计，Qwen执行具体实现 |

### 3.2 上下文窗口溢出处理流程

```
检测到任务可能超出模型上下文窗口
    │
    ▼
【检查】当前模型的上下文窗口大小是否足够？
    ├── 是 → 正常处理，但监控Token使用率（<80%）
    └── 否 → 进入溢出处理流程：
         │
         ▼
    1. 识别任务中"必须加载"的文件（核心需求相关）
    2. 识别"可选参考"的文件（依赖关系、历史决策）
    3. 优先加载必须文件，可选文件通过路径引用
    4. 如仍超出窗口 → 拆分为多个子任务
    5. 每个子任务使用独立的对话会话
```

### 3.3 模型上下文窗口对比表

| 模型 | 最大上下文窗口 | 安全使用上限（80%） | 适用任务规模 |
|------|--------------|-------------------|-------------|
| Gemma（oMLX） | ~8K tokens | ~6K tokens | 单文件，<50行代码 |
| Qwen（oMLX） | ~32K-128K tokens | ~25K-100K tokens | 单模块，<200行代码 |
| Claude（云端） | ~200K tokens | ~160K tokens | 跨模块架构设计 |
| GPT（云端） | ~128K tokens | ~100K tokens | 具体实现细节优化 |

## 4. 网络状态自适应策略

### 4.1 网络状态检测与模型切换

```
检测到网络状态变化
    │
    ├── 有网络 → 允许使用云端模型（Claude/GPT）
    │   │
    │   ▼
    │  【检查】网络质量是否足够？
    │   ├── 良好（<500ms延迟） → 正常使用云端模型
    │   └── 较差（>500ms延迟） → 优先本地模型，仅在必要时使用云端
    │
    └── 无网络 → 强制切换到本地模型
        │
        ▼
    【检查】任务复杂度是否超出本地模型能力？
        ├── 简单/中等 → 使用Gemma或Qwen正常处理
        └── 复杂 → 
             ├── 方案A：使用Qwen最大上下文模式（128K）
             └── 方案B：将任务拆分为多个本地可处理子任务
```

### 4.2 网络状态与模型选择对照表

| 网络状态 | 简单任务 | 中等任务 | 复杂任务 |
|---------|---------|---------|---------|
| **有网络（良好）** | Gemma（本地） | Qwen（本地）或 Claude/GPT | Claude/GPT（云端） |
| **有网络（较差）** | Gemma（本地） | Qwen（本地）优先，Claude/GPT备用 | 优先Qwen，必要时Claude/GPT |
| **无网络** | Gemma（本地） | Qwen（本地，最大上下文模式） | Qwen（拆分策略）或 等待网络恢复 |

## 5. 多模型对比矩阵

### 5.1 综合评分表（满分5分）

| 评估维度 | Gemma（本地） | Qwen（本地） | Claude（云端） | GPT（云端） |
|---------|-------------|-------------|---------------|------------|
| **响应速度** | 5（<100ms） | 4（100-500ms） | 2（500ms-2s） | 3（300ms-1.5s） |
| **代码质量** | 3（简单到中等） | 4（中等复杂度） | 5（复杂架构） | 4（具体实现） |
| **上下文窗口** | 2（8K） | 4（32-128K） | 5（200K+） | 4（128K） |
| **离线可用性** | 5 | 5 | 0 | 0 |
| **成本** | 5（免费本地） | 5（免费本地） | 2（按量计费） | 3（按量计费，通常较低） |
| **综合推荐度** | 简单任务首选 | 中等任务首选 | 复杂架构首选 | API适配首选 |

### 5.2 成本-质量权衡决策表

| 优先级 | 推荐模型组合 | 说明 |
|--------|-------------|------|
| **成本优先**（个人项目/原型） | 100%本地模型（Gemma+Qwen） | 零云端成本，适合开发和调试阶段 |
| **质量优先**（商业项目/发布前） | 本地模型开发 + Claude/GPT审查 | 用本地模型完成80%工作，云端模型做最后20%的质量把关 |
| **平衡模式**（默认推荐） | 简单/中等任务本地，复杂任务云端 | 根据任务复杂度自动切换，兼顾成本和质量 |

## 6. 模型切换决策流程（TRAE自动执行）

### 6.1 自动决策算法

当用户提出开发任务时，TRAE按以下流程自动选择模型：

```
输入：用户的开发任务描述
    │
    ▼
Step 1: 估算任务复杂度（基于关键词和文件数量）
    │   - 简单：UI组件、变量命名、注释 → Gemma
    │   - 中等：单模块开发、数据模型、测试 → Qwen
    │   - 复杂：跨模块架构、安全审计 → Claude/GPT
    │
    ▼
Step 2: 检查网络状态
    │   - 有网络：允许云端模型候选
    │   - 无网络：排除云端模型，仅使用本地模型
    │
    ▼
Step 3: 检查成本约束（来自PROJECT_CONTEXT.md）
    │   - scope="mvp"或"small"：优先本地模型
    │   - scope="full"：允许合理使用云端模型
    │
    ▼
Step 4: 输出推荐模型及理由（向用户展示）
    │   "根据任务复杂度[简单/中等/复杂]和网络状态[有/无]，
    │    推荐使用 [Gemma/Qwen/Claude/GPT]。"
    │
    ▼
Step 5: 等待用户确认（或根据预设策略自动执行）
```

### 6.2 手动覆盖模型选择

用户可随时通过指令覆盖TRAE的自动选择：

| 用户指令 | TRAE行动 |
|---------|---------|
| "用Gema处理" | 强制使用Gemma，即使任务超出其能力范围会发出警告 |
| "用Qwen处理" | 强制使用Qwen，即使任务超出其能力范围会发出警告 |
| "用Claude处理" | 强制使用Claude，检查网络状态后执行 |
| "用GPT处理" | 强制使用GPT，检查网络状态后执行 |
| "你决定用哪个" | 按上述自动决策算法执行 |
