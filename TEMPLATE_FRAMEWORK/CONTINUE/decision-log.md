# 决策日志模板（Decision Log Template）

> 此文件为 CONTINUE/ 目录下的决策日志模板。TRAE 在遇到重要技术决策时，应基于此格式生成具体的决策日志文件（命名规则：`YYYY-MM-DD-[module]-[short-title].md`）。

---

## 使用说明

### 何时使用此模板

当用户在开发过程中做出以下类型的决策时，TRAE 应基于此模板生成具体的决策日志文件：

1. **技术选型** — 选择框架、库、工具或技术方案
2. **架构决策** — 选择架构模式、数据流设计、模块划分方式
3. **需求变更** — 新增、删除或修改功能需求
4. **UI/UX 设计决策** — 选择布局、交互方式、视觉风格
5. **性能优化方案** — 选择特定的优化策略或技术方案

### 文件命名规则

```
CONTINUE/decision-log/YYYY-MM-DD-[module]-[short-title].md
```

| 部分 | 说明 | 示例 |
|------|------|------|
| `YYYY-MM-DD` | 决策日期 | 2025-01-15 |
| `[module]` | 相关模块名称（小写，连字符分隔） | login、home-screen、data-persistence |
| `[short-title]` | 简短描述（小写，连字符分隔） | use-swiftui instead-of-uikit、choose-local-first-storage |

**示例文件名：**
- `2025-01-15-login-use-swiftui-auth.md`
- `2025-02-20-home-screen-choose-mvvm-architecture.md`
- `2025-03-10-data-persistence-choose-local-first.md`

## 示例：技术选型决策日志

### 文件路径：`CONTINUE/decision-log/2025-01-15-login-use-swiftui-auth.md`

```markdown
## Decision Log: 登录模块使用 SwiftUI Auth 而非 Firebase

- **日期：** 2025-01-15
- **模块：** 登录（login）
- **决策类型：** 技术选型
- **提出者：** 人类用户

### 背景

登录模块需要实现用户认证功能。最初考虑使用 Firebase Auth，但考虑到项目要求本地优先（local-first）架构原则，以及希望减少对外部服务的依赖，需要重新评估认证方案。

### 选项 A：Firebase Auth

**描述：** 使用 Google Firebase 提供的认证服务，支持邮箱/密码、Google OAuth、Apple OAuth。

**优点：**
- 成熟的认证方案，安全性经过大规模验证
- 支持多种登录方式（邮箱、OAuth）
- Firebase 提供用户管理后台，方便运营

**缺点：**
- 依赖外部云服务，违反 local-first 原则
- 需要网络连接才能使用
- 增加项目对外部服务的依赖

### 选项 B：本地密钥链 + SwiftUI Auth（选择）

**描述：** 使用 iOS 原生 Keychain 存储认证凭证，结合 SwiftUI 的 AuthenticationServices 框架实现 Apple Sign-In。

**优点：**
- 完全本地化，不依赖外部服务
- 符合 local-first 架构原则
- 利用 iOS 原生安全机制（Keychain）

**缺点：**
- 仅支持 Apple Sign-In，不支持邮箱/密码登录
- 设备更换后需要重新认证

### 最终决定：选择了本地密钥链 + SwiftUI Auth

**原因：** 项目核心原则是 local-first，用户数据不应依赖外部服务。虽然 Firebase Auth 功能更丰富，但对于 MVP 阶段来说，Apple Sign-In 已足够。未来如需扩展邮箱登录，可在 ARCHIVE/决策日志中记录扩展决策。

### 影响范围

**受影响的文件：**
- `PRD/login.md` — 认证方案从 Firebase 改为本地 Keychain
- `Sources/Auth/AppleSignInView.swift` — 新建，实现 Apple Sign-In UI
- `Sources/Auth/KeychainManager.swift` — 新建，管理密钥链存取

**受影响的模块：**
- 登录（login）— 认证方案变更

### 后续回顾日期：2025-02-14（30天后）

**回顾时检查：**
- [ ] Apple Sign-In 是否满足用户认证需求？
- [ ] 是否需要增加邮箱/密码登录方式？
- [ ] Keychain 存储是否满足安全要求？

### 历史修订记录

| 日期 | 修订内容 | 修订人 |
|------|---------|--------|
| 2025-01-15 | 初始创建，确定使用本地方案 | 人类用户 |
```

## TRAE 自动生成决策日志的规则

### 触发条件

TRAE 在以下对话场景中应**自动检测并生成决策日志**：

| 触发信号 | 示例用户输入 | 行动 |
|---------|-------------|------|
| "我们改用..." | "我们改用本地存储吧" | 生成决策日志，记录从云服务到本地的变更 |
| "改为使用..." | "改为使用 Keychain 存储" | 生成决策日志，记录方案变更 |
| "选择...而不是..." | "我选择 MVVM 而不是 MVC" | 生成决策日志，记录架构选型 |
| "不用 XX 了" | "不用 Firebase 了，用本地方案" | 生成决策日志，记录取消的方案 |
| "这个方案有问题" | "Firebase 太慢了，换个方案" | 生成决策日志，记录问题和新方案 |

### 自动生成流程

1. **检测信号：** 识别用户输入中的决策相关关键词
2. **提取信息：** 从对话上下文提取当前模块、原方案、新方案
3. **生成模板：** 基于本模板生成具体决策日志文件
4. **确认：** 向用户展示生成的决策日志，请求确认或修改
5. **存储：** 保存到 `CONTINUE/decision-log/YYYY-MM-DD-[module]-[short-title].md`
6. **索引：** 在 PROJECT_CONTEXT.md 的"技术架构概述"中引用关键决策

### 注意事项

- TRAE 不应为简单的 bug 修复生成决策日志（除非涉及架构影响）
- 如果对话中未做出明确决策，TRAE 不应强行生成
- 用户可以说"不需要记录这个决策"来跳过自动生成
