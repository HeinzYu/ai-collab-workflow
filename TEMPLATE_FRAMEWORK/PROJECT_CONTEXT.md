## 9. 长期项目记忆衰减策略

### 9.1 问题背景

AI Agent（如 TRAE）在每次新对话中无法记住之前的对话内容。对于持续3个月、半年甚至一年的项目，如果没有有效的"外部记忆"机制，AI 会在每次新对话开始时丢失大量上下文信息。

这就是"记忆衰减"问题：项目知识存在于文件系统中，但 AI 无法自动读取所有历史文档。

### 9.2 三层记忆架构

```
外部记忆（文件系统 — AI 可读取）
├── 永久层：AGENTS.md, PROJECT_CONTEXT.md, MODEL_CONFIG.md
│   └── 每次对话自动加载，无需人工干预
├── 稳定层：已完成的 PRD, CONTINUE/总结, ARCHIVE/摘要
│   └── 通过路径按需加载，不自动全量读取
└── 索引层：REFERENCES.md, STRUCTURE.md
    └── 提供文件定位，AI 根据索引找到需要的内容

内部记忆（对话上下文 — 单次会话有效）
├── 当前对话加载的内容
│   └── 永久层 + 当前模块 PRD + 相关总结
└── 随对话轮数增加而累积的中间状态
    └── 会话结束后丢失（除非写入外部记忆）

人类记忆（经验与直觉 — 不可替代）
├── 对项目的整体理解
├── 技术决策的背景和权衡
└── AI 无法完全替代的判断力
```

### 9.3 记忆衰减曲线与补偿策略

```
项目时间（月）    AI 对项目理解的准确度
0-1              ████████████ 高（有完整 PRD + 开发日志）
1-2              ██████████   中高（月度摘要补偿部分衰减）
2-3              ████████     中（需要季度审计恢复上下文）
3+               █████        低（无补偿机制）→ 必须依赖外部记忆
```

**核心原则：通过完善的外部记忆（文件系统文档），使 AI 在项目任何阶段都能恢复到接近初始的理解水平。**

### 9.4 恢复策略：新对话时的上下文重建

每次开启新对话时，TRAE 按以下顺序重建项目理解：

```
新对话开始
    │
    ▼
第1步：加载永久层（自动）
  ├── AGENTS.md → 理解项目规则和工作流
  ├── PROJECT_CONTEXT.md → 了解项目目标和当前状态
  └── MODEL_CONFIG.md → 确认可用模型配置
    │
    ▼
第2步：定位当前工作模块
  ├── 读取 PROJECT_CONTEXT.md 中的"已开发模块"列表
  └── 识别最后开发的模块（按里程碑或时间判断）
    │
    ▼
第3步：加载工作层（按需）
  ├── 读取当前模块的 PRD（了解需求）
  └── 读取该模块最近的 CONTINUE/总结（了解进度）
    │
    ▼
第4步：必要时加载稳定层
  ├── 读取相关模块的 PRD（理解依赖关系）
  └── 读取 ARCHIVE/中的月度摘要（了解历史决策）
    │
    ▼
第5步：确认理解，开始工作
  └── 向用户展示当前项目状态摘要（来自 PROJECT_CONTEXT.md）
```

### 9.5 记忆衰减预警机制

当项目运行超过1个月时，TRAE 在新对话中自动检查：

| 检查项 | 预警条件 | 行动 |
|-------|---------|------|
| 月度摘要是否存在 | 项目 >30天但 ARCHIVE/ 无月度文件 | 提醒用户执行首次月维护 |
| 模块状态是否更新 | PROJECT_CONTEXT.md 中"最后更新" >7天 | 提醒用户确认项目状态 |
| DevLog 是否连续 | DEVLOG/ 中存在超过14天的空白期 | 提醒用户检查项目是否有中断 |
| 模型配置是否过时 | MODEL_CONFIG.md 中配置超过30天未审查 | 提醒用户检查本地模型版本 |

### 9.6 跨月项目恢复协议

当距离上次开发超过7天时，开启新对话前 TRAE 必须执行：

1. **读取最近3个月的 ARCHIVE/月度摘要**（如有）
2. **列出所有已完成模块及其总结**（来自 PROJECT_CONTEXT.md 和 CONTINUE/）
3. **向用户展示"上次开发时项目状态"**，包括：
   - 最后完成的模块名称和日期
   - 当前正在开发的模块（如有）
   - 已知问题和待办事项
4. **确认用户理解当前状态**后，才开始新开发任务

### 9.7 最佳实践

1. **每月至少生成一次月度摘要** — 这是防止记忆衰减的最重要机制
2. **完成模块后立即生成总结** — CONTINUE/ 总结是跨会话恢复的关键
3. **保持 PROJECT_CONTEXT.md 更新** — 模块状态和里程碑必须反映实际进度
4. **季度审计不可跳过** — 超过90天的项目，季度审计是恢复上下文的最有效方式
5. **人类定期回顾** — 即使 AI 有外部记忆，人类也应定期（至少每月）回顾项目文档

## 10. 项目健康仪表板（Project Health Dashboard）

项目健康仪表板是一个统一的视图，整合维护状态、开发进度、风险预警和系统健康度四个维度。它让开发者和 AI 在单页内掌握项目整体状态，快速识别需要关注的问题。

### 10.1 仪表板结构

```
项目健康仪表板 — [项目名称]
├── A. 维护状态（Maintenance Status）
│   ├── 上次周维护：[YYYY-MM-DD] — [正常/逾期]
│   ├── 上次月维护：[YYYY-MM-DD] — [正常/逾期]
│   └── 上次季维护：[YYYY-MM-DD] — [正常/逾期]
│
├── B. 开发进度（Development Progress）
│   ├── 总模块数：[X] 个
│   ├── 已完成：[Y] 个（[Z]%）
│   ├── 开发中：[A] 个
│   └── 未开始：[B] 个（[C]%）
│
├── C. 风险预警（Risk Alerts）
│   ├── 逾期模块：[列出超过14天未完成的开发中模块]
│   ├── 记忆衰减预警：[列出触发预警的检查项]
│   └── 技术风险：[列出当前未解决的技术风险]
│
└── D. 系统健康度（System Health）
    ├── 核心文件完整性：[AGENTS.md ✓ / PROJECT_CONTEXT.md ✓ / MODEL_CONFIG.md ✓]
    ├── 决策日志数量：[X] 个（最近30天）
    └── Token 使用趋势：[上升/稳定/下降]（最近4周）
```

### 10.2 维护状态详解

| 检查项 | 正常范围 | 预警条件 | 行动 |
|-------|---------|---------|------|
| **周维护** | 距上次 ≤7天 | 距上次 >7天 | 提醒用户执行周维护（§12.1） |
| **月维护** | 距上次 ≤30天 | 距上次 >30天 | 提醒用户执行月维护（§12.2） |
| **季维护** | 距上次 ≤90天 | 距上次 >90天 | 提醒用户执行季维护（§12.3） |

### 10.3 开发进度可视化

```
项目整体完成度：[████████░░] XX%

已完成模块（Y/[X]）：
  ✅ [模块名1] — 完成日期：YYYY-MM-DD
  ✅ [模块名2] — 完成日期：YYYY-MM-DD

开发中模块（A/[X]）：
  🔄 [模块名3] — 开始日期：YYYY-MM-DD，已开发 [N] 天
  🔄 [模块名4] — 开始日期：YYYY-MM-DD，已开发 [M] 天

未开始模块（B/[X]）：
  ⏸️ [模块名5] — 优先级：[高/中/低]
  ⏸️ [模块名6] — 优先级：[高/中/低]
```

**进度健康标准：**
- **优秀：** 完成度 >70%，无逾期模块，无预警
- **良好：** 完成度 40-70%，≤1个逾期模块，≤2个预警
- **警告：** 完成度 20-40%，>1个逾期模块，或 >2个预警
- **危险：** 完成度 <20%，或存在超过30天的"僵尸模块"（长期未开发也未调整优先级）

### 10.4 风险预警机制

| 风险类型 | 触发条件 | 严重等级 | 行动 |
|---------|---------|---------|------|
| **模块逾期** | 开发中状态超过14天无更新 | 🟡 警告 | 提醒用户确认进度或调整状态 |
| **记忆衰减** | 月度摘要缺失（项目 >30天） | 🟡 警告 | 提醒执行首次月维护 |
| **记忆衰减** | 模块状态 >7天未更新 | 🟠 中等 | 提醒确认项目状态 |
| **记忆衰减** | DevLog >14天空白期 | 🟠 中等 | 提醒检查项目是否中断 |
| **记忆衰减** | 模型配置 >30天未审查 | 🟡 警告 | 提醒检查本地模型版本 |
| **僵尸模块** | 未开始状态超过30天 | 🔴 严重 | 建议调整优先级或从项目中移除 |
| **上下文溢出** | 单次对话 Token >安全上限80% | 🟠 中等 | 建议开启新对话，拆分任务 |

### 10.5 系统健康度检查

| 检查项 | 正常状态 | 异常处理 |
|-------|---------|---------|
| **核心文件完整性** | AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md 均存在且可读取 | 检查文件是否被意外删除或移动 |
| **决策日志数量** | 最近30天至少有2-3个决策日志（视项目复杂度） | 提醒用户补充重要决策记录 |
| **Token 使用趋势** | 稳定或下降（通过优化策略降低消耗） | 检查是否有异常高消耗的对话 |
| **CONTINUE/完整性** | 每个已完成模块都有对应的总结文件 | 提醒为缺失的模块生成总结 |
| **ARCHIVE/完整性** | 每月至少1个月度摘要（项目 >30天时） | 提醒执行补做月维护 |

### 10.6 仪表板更新频率

| 触发条件 | 行动 |
|----------|------|
| **每次新对话开始** | TRAE 自动生成并展示当前仪表板状态（前3轮内） |
| **模块完成时** | 更新开发进度部分，标记模块为"已完成" |
| **执行维护时** | 更新维护状态部分，记录最新维护日期 |
| **风险预警触发时** | 在仪表板中显示最新预警，记录处理状态 |
| **每月1日** | 自动生成上月健康报告（可配置为自动化任务） |
