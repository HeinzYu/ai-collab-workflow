# AGENTS.md — 全局规则

> AI Agent 在此项目中必须遵循的全局行为准则。此文件为始终详细的核心文档，不可展开或简化。
>
> **兼容性声明：** 本文档默认面向 **TRAE Agent** 编写，但架构设计为 **工具无关**（Tool-Agnostic）。所有规则、模板和工作流均可无缝适配 Claude Code、Cursor、Codex、Gemini CLI 等任何 AI Coding Agent。只需将文档中的 "TRAE" 替换为对应 Agent 名称即可。详见 README.md 中的设计哲学。
>
> **当前版本：** 1.2 | **最后更新：** 2026-06-28 | **适用场景：** Apple App / Web / Backend / Game 项目

## 1. 项目初始化指令

当用户描述新项目时，根据项目类型判断生成的模块：

### 完整商业项目（默认）
触发指令示例："这是一个完整的 Apple App 项目，包含登录、首页、设置等功能"

执行动作：
1. 生成完整的目录结构（见 PROJECT_CONTEXT.md）
2. 初始化所有核心文件（AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md、REFERENCES.md、TEMPLATE_GUIDE.md）
3. 初始化所有模块占位符（PRD/、ARCHITECTURE/、TESTING/、DEPLOYMENT/、DEVLOG/、CONTINUE/）
4. 在 REFERENCES.md 中记录项目来源文档（如有）

### 小型工具项目
触发指令示例："这是一个简单的工具 App，只需要登录和首页功能"

执行动作：
1. 生成核心文件 + 选定的模块占位符
2. 跳过非必要模块（如 TESTING/、DEPLOYMENT/）
3. 在 PROJECT_CONTEXT.md 的 `scope` 字段中标记为 "small"

### 最小可行项目（MVP）
触发指令示例："这是一个原型验证，只需要一个页面"

执行动作：
1. 仅生成核心文件 + PRD/ 下对应功能文档的占位符
2. 其他模块全部跳过，在 PROJECT_CONTEXT.md 中标记为 "mvp"

## 3. 模块总结机制（CONTINUE/）

每完成一个功能模块的开发，必须生成一份 `*-summary.md` 文件：

触发条件：
- 用户完成一个 PRD 中定义的功能模块
- 用户手动要求生成总结

执行动作：
1. 读取 `PRD/<module>.md` 中的需求描述
2. 读取 `DEVLOG/` 中该模块开发期间的日志
3. 生成 `CONTINUE/<module>-summary.md`，包含：
   - 功能概述（2-3句话）
   - 关键技术决策
   - 已知问题和待办事项
   - 相关文件路径索引

---

## 4. 开发日志机制（DEVLOG/）

每日开发活动记录：

触发条件：
- 每次开启新的对话会话（由 TRAE 自动生成）
- 用户手动添加日志条目

执行动作：
1. 检查 `DEVLOG/` 中是否已有当日文件（格式：YYYY-MM-DD.md）
2. 若无，创建新文件，包含日期、会话摘要、今日任务列表
3. 若有，追加新的日志条目

## 5. 大文件处理规则（混合策略）

### 文档类（doc/PDF）
1. TRAE 读取原文 → 生成简短 Markdown 摘要（写入 REFERENCES.md）
2. 后续对话只引用摘要，需要细节时通过路径读取原文
3. 避免上下文溢出

### Figma 设计稿
1. TRAE 通过 Figma MCP 按页面/组件拆分导出
2. 每个页面生成独立文档（`ASSETS/figma/<page-name>.md`）
3. 在 REFERENCES.md 中建立索引，TRAE 按需读取对应页面

### 处理流程
1. 用户提交大文件时，TRAE 自动识别类型（doc/PDF/Figma）
2. 按对应策略处理，不直接加载全文到上下文
3. 在 REFERENCES.md 中记录：文件类型、路径、摘要、关键信息点

## 6. 本地模型配置引用

每次对话开始前，TRAE 必须读取 `MODEL_CONFIG.md`：
1. 确认当前可用的本地模型（Gemma/Qwen）及其配置
2. 根据任务复杂度选择合适模型
3. 遵守上下文窗口限制，超出时自动拆分任务

---

## 7. 对话管理策略

### 上下文溢出预防
- 单次对话不超过模型最大 token 的 80%
- 超过限制时，优先保留 AGENTS.md、PROJECT_CONTEXT.md、当前模块的 PRD
- 历史文档通过 REFERENCES.md 引用，不加载到上下文

### 任务拆分规则
- 单个功能模块的开发拆分为不超过 10 个子任务
- 每个子任务的输出文件不超过 500 行代码或 20 个文件
- 复杂功能先写 PRD，再按 PRD 逐步实现

### 错误处理
- 模型返回错误或超时：记录到 DEVLOG/，重试一次后通知用户
- 文件读取失败：检查路径是否正确，尝试从 REFERENCES.md 重新定位

## 8. 文件读写规范

### 读取顺序
1. AGENTS.md（始终）
2. PROJECT_CONTEXT.md（首次对话）
3. MODEL_CONFIG.md（每次对话前）
4. 当前模块的 PRD（开发该模块时）

### 写入顺序
1. 更新对应模块的 PRD（需求变更时）
2. 追加 DEVLOG/ 日志（每次对话后）
3. 更新 CONTINUE/ 总结（模块完成后）
4. 更新 REFERENCES.md（新增大文件时）

## 11. Context 生命周期分层

AI 开发中，Context（上下文/记忆）不是单一概念，而是有明确生命周期的多层结构。不同层级的文档具有不同的持久性和访问策略。

### 11.1 五层 Context 架构

```
永久层（Permanent）
  ├── AGENTS.md — 始终加载，永不删除
  ├── PROJECT_CONTEXT.md — 项目核心信息，长期保留
  └── MODEL_CONFIG.md — 模型配置，跨会话有效

稳定层（Stable）
  ├── PRD/ 下已完成的模块文档
  ├── CONTINUE/ 下的所有总结文件
  └── ARCHIVE/ 下的月度/季度摘要

工作层（Working）
  ├── 当前正在开发的 PRD 模块
  └── 最近 7 天的 DevLog

临时层（Temporary）
  ├── 最近 30 天的 DevLog
  └── 当前会话的中间产物

可弃层（Disposable）
  ├── 超过 30 天的 DevLog
  └── 已完成且无后续引用的中间文件
```

### 11.2 各层级管理策略

| 层级 | 保留期限 | 自动加载 | 清理策略 | 示例文件 |
|------|---------|---------|---------|---------|
| 永久层 | 项目全周期 | 每次对话 | 人类手动决定删除 | AGENTS.md, PROJECT_CONTEXT.md |
| 稳定层 | 项目全周期 | 按需（通过路径） | 人类手动决定归档 | CONTINUE/*-summary.md, 已完成 PRD |
| 工作层 | 当前开发周期 | 当前会话 | 模块完成后降级为稳定层 | 正在开发的 PRD, 最近 DevLog |
| 临时层 | 30天 | 不自动加载 | 月度清理时降级为可弃层 | 最近30天 DevLog |
| 可弃层 | 无限制 | 不加载 | 季度清理时物理删除 | 超过30天的 DevLog |

### 11.3 Context 层级转换规则

```
开发中模块（工作层）
    ↓ 模块完成并生成总结
已完成模块（稳定层）
    ↓ 30天后未再引用
临时层 → 月度清理时标记为可弃
    ↓ 季度清理
物理删除（或归档到 ARCHIVE/）
```

### 11.4 TRAE 自动管理规则

1. **模块完成时**：将 PRD 从工作层标记为稳定层（在 PROJECT_CONTEXT.md 中更新状态）
2. **对话恢复时**：优先加载永久层 + 稳定层，按需加载工作层
3. **月度维护时**：将超过30天的临时层 DevLog 标记为可弃
4. **季度维护时**：物理删除或归档超过90天的可弃层文件

## 12. 周期性维护检查清单

长期项目（超过1个月）需要周期性的维护和健康检查，确保 AI 对项目上下文的理解不会衰减。

### 12.1 每周维护（推荐每7天执行一次）

触发指令："执行本周项目维护"

**检查清单：**
- [ ] 确认 DEVLOG/ 中每天都有开发记录
- [ ] 检查是否有未完成的 PRD 模块长期处于"开发中"状态（超过14天）
- [ ] 确认最近完成的模块已生成 CONTINUE/ 总结
- [ ] 检查 PROJECT_CONTEXT.md 中的里程碑状态是否与实际进度一致
- [ ] 生成周总结文件 `DEVLOG/weekly-YYYY-WXX.md`（包含本周完成内容、下周计划）

**TRAE 自动执行：**
1. 读取 PROJECT_CONTEXT.md 中所有模块状态
2. 对比 PRD/ 目录下的实际文件
3. 标记状态不一致的模块
4. 生成周总结并追加到 DEVLOG/

### 12.2 每月维护（推荐每30天执行一次）

触发指令："执行本月项目归档"

**检查清单：**
- [ ] 清理临时层：将超过30天的 DevLog 标记为可弃
- [ ] 检查是否有模块长期处于"未开始"状态（超过30天），评估是否需要调整优先级
- [ ] 检查 MODEL_CONFIG.md 中的模型配置是否仍适用（本地模型是否有升级）
- [ ] 生成月度摘要 `ARCHIVE/monthly-YYYY-MM.md`，包含：
  - 本月完成的功能模块
  - 技术决策记录
  - 已知问题和待办事项
  - 下月计划
- [ ] 更新 PROJECT_CONTEXT.md 中的里程碑时间线

**TRAE 自动执行：**
1. 扫描 DEVLOG/ 中所有文件，找出超过30天的条目
2. 将超出条目标记为可弃（不立即删除，保留到季度清理）
3. 读取所有已完成模块的 PRD 和总结
4. 生成月度摘要文件到 ARCHIVE/
5. 更新 PROJECT_CONTEXT.md 中的模块状态和里程碑

### 12.3 每季度维护（推荐每90天执行一次）

触发指令："执行季度项目审计"

**检查清单：**
- [ ] 物理删除或归档超过90天的可弃层 DevLog 文件
- [ ] 审查项目 scope，评估是否需要调整（full/small/mvp）
- [ ] 检查 MODEL_CONFIG.md：本地模型是否有新版本，配置是否需要更新
- [ ] 评估项目整体健康度（进度、风险、技术债务）
- [ ] 生成季度审计报告 `ARCHIVE/quarterly-YYYY-QX.md`

**TRAE 自动执行：**
1. 清理 DEVLOG/ 中超过90天的临时层 DevLog（物理删除或移动到 ARCHIVE/）
2. 读取 PROJECT_CONTEXT.md 评估项目 scope 是否仍然合理
3. 检查 MODEL_CONFIG.md 中的配置参数是否需要调整
4. 生成季度审计报告到 ARCHIVE/

### 12.4 维护触发方式

| 方式 | 说明 |
|------|-----|
| 手动触发 | 用户对 TRAE 说"执行本周/月/季度维护" |
| 自动化提醒 | TRAE 在 PROJECT_CONTEXT.md 中记录上次维护日期，下次对话时自动提醒 |
| 里程碑触发 | 完成重大模块后自动执行一次维护检查 |

### 12.5 维护日期追踪

在 PROJECT_CONTEXT.md 中增加以下字段：

```
- **上次周维护：** [YYYY-MM-DD]
- **上次月维护：** [YYYY-MM-DD]
- **上次季维护：** [YYYY-MM-DD]
```

每次执行对应级别的维护后，TRAE 自动更新这些日期。超过7天未执行周维护、超过30天未执行月维护、超过90天未执行季维护时，TRAE 在每次对话开始时提醒用户。

## 13. 对话策略（Conversation Strategy）

AI 开发中，单次对话（Conversation）不是"无限聊天"，而是有明确职责和边界的独立工作单元。合理控制单次对话的范围，可以显著提高输出质量和系统稳定性。

### 13.1 单次对话的职责定义

| 对话类型 | 核心职责 | 不应包含的内容 |
|----------|---------|---------------|
| **初始化对话** | 创建项目结构、生成核心文件、确定 scope | 实际功能开发 |
| **模块开发对话** | 完成单个 PRD 中定义的功能模块 | 跨多个不相关模块的开发 |
| **调试对话** | 修复单个错误或问题 | 同时处理多个不相关的 bug |
| **重构对话** | 对单个文件或模块进行代码重构 | 全项目范围的重构（需拆分） |
| **审查对话** | Code Review、架构评审 | 开发与审查同时进行 |
| **设计对话** | UI/UX 设计、交互原型 | 设计与后端开发同时进行 |
| **测试对话** | 编写和执行测试用例 | 新功能开发与测试混在一起 |

### 13.2 推荐限制

| 约束类型 | 推荐上限 | 说明 |
|----------|---------|------|
| **最大对话轮数** | 30 轮（一问一答 = 1轮） | 超过后上下文质量下降，应开启新对话 |
| **最大 Token 使用** | 模型安全上限的 80% | 超过后自动触发上下文溢出处理（AGENTS.md §7） |
| **最大开发时间** | 单次对话不超过 2 小时 | 长时间开发应拆分为多个子对话 |
| **最大文件变更** | 单次对话不超过 20 个文件 | 超过后应拆分为多个子任务 |
| **最大代码行数** | 单次对话不超过 500 行新增代码 | 超过后应拆分为多个子任务 |

### 13.3 对话生命周期管理

```
对话开始
    │
    ▼
第1轮：读取永久层 + 定位当前模块（§8 文件读写规范）
    │
    ▼
第2-N轮：执行具体开发任务
    │   - 每 10 轮检查一次上下文使用率（MODEL_CONFIG.md §7.3）
    │   - 每完成一个子任务，在 DEVLOG/ 中记录进度
    │
    ▼
最后 3 轮：
    ├── 生成/更新 CONTINUE/ 总结（§3）
    ├── 追加 DEVLOG/ 日志（§4）
    └── 更新 PROJECT_CONTEXT.md 中的模块状态
    │
    ▼
对话结束 → 标记为"已完成"或"需继续"
```

### 13.4 New Chat / Continue / Summary 决策时机

| 信号 | 应执行操作 |
|------|-----------|
| 上下文使用率达到 80% | **开启新对话（New Chat）**，保留核心文件，历史通过路径引用 |
| 当前模块开发完成 | **生成总结（Summary）**到 CONTINUE/，标记为"已完成" |
| 模块开发中断 >7天 | **开启新对话（New Chat）**，先加载上次总结再开始（PROJECT_CONTEXT.md §9.6 跨月恢复协议） |
| 连续开发 >2小时或 >30轮 | **生成中间总结**到 CONTINUE/，开启新对话继续剩余工作 |
| 遇到无法解决的错误 | **记录到 DEVLOG/**，开启新对话从不同角度尝试 |
| 需要跨模块协作 | **开启新对话**，分别处理各模块后通过 PROJECT_CONTEXT.md 整合 |

### 13.5 对话质量检查清单（每 10 轮执行一次）

- [ ] 当前上下文使用率是否 <80%？
- [ ] 当前对话是否仍聚焦在单一职责上？
- [ ] 是否有明确的下一步行动？
- [ ] 重要决策是否已记录到 DEVLOG/ 或 CONTINUE/？
- [ ] 文件变更数量是否仍在可接受范围内（≤20个文件）？

## 14. 决策日志管理（Decision Log Management）

在 AI 辅助开发中，每一次重要的技术选择、架构决策或需求变更都应该被记录。这些"决策日志"是长期项目中最有价值的资产之一——它们让 AI 在未来理解"为什么这样做"，而不仅仅是"做了什么"。

### 14.1 决策日志格式规范

每个决策日志条目必须包含以下字段：

```markdown
## Decision Log: [简短标题]
- **日期：** YYYY-MM-DD
- **模块：** 相关模块名称（如"登录"、"首页"）
- **决策类型：** 架构 / 技术选型 / 需求变更 / UI设计 / 性能优化 / 其他
- **背景：** 为什么需要做出这个决策（2-3句话）
- **选项 A：** [方案描述] — 优点 / 缺点
- **选项 B：** [方案描述] — 优点 / 缺点
- **[可选] 更多选项...**
- **最终决定：** 选择了哪个方案及原因（2-3句话）
- **影响范围：** 受影响的文件或模块列表
- **后续回顾日期：** YYYY-MM-DD（建议30天后回顾此决策）
```

### 14.2 何时必须记录决策日志

| 场景 | 是否必须记录 |
|------|-------------|
| 选择技术栈或框架版本 | ✅ 必须 |
| 架构模式选择（MVVM/Clean Architecture等） | ✅ 必须 |
| 数据持久化方案选择 | ✅ 必须 |
| UI/UX 设计决策（布局、交互方式） | ✅ 必须 |
| 需求变更（新增/删除/修改功能） | ✅ 必须 |
| 性能优化方案选择 | ⚠️ 推荐（影响较大的时） |
| 第三方库或 API 选择 | ⚠️ 推荐 |
| 简单的 bug 修复 | ❌ 不需要（除非涉及架构影响） |

### 14.3 决策日志存储位置

```
PROJECT/
├── CONTINUE/
│   ├── module-summary.md          # 模块总结（已有）
│   └── decision-log/              # [新增] 决策日志目录
│       ├── YYYY-MM-DD-[module]-[short-title].md
│       └── ...
├── ARCHIVE/
│   ├── monthly-YYYY-MM.md         # 月度摘要（已有）
│   └── quarterly-YYYY-QX.md       # 季度审计（已有）
└── PROJECT_CONTEXT.md             # 在"技术架构概述"中引用关键决策
```

### 14.4 TRAE 自动记录规则

TRAE 在以下情况应**自动生成决策日志**：
1. 用户说"我们改用..."、"改为使用..."、"选择...而不是..."
2. 用户要求创建新的架构模块或数据模型
3. 用户要求引入第三方依赖或框架
4. 上下文使用率达到 70% 时，TRAE 主动检查是否有未记录的决策

### 14.5 决策回顾机制

| 回顾时机 | 行动 |
|----------|------|
| **30天回顾** | 检查决策是否仍适用，是否需要调整 |
| **模块完成时** | 回顾该模块相关的所有决策，确认无遗漏问题 |
| **月度维护时** | 扫描所有未回顾的决策日志，标记需要审查的条目 |
| **季度审计时** | 全面评估历史决策，更新 PROJECT_CONTEXT.md 中的架构概述 |

### 14.6 决策日志的价值

1. **AI 理解加速：** 新对话通过阅读决策日志，可以在3轮内恢复到与老对话相同的理解水平
2. **避免重复讨论：** 不需要每次都重新讨论"为什么选择这个方案"
3. **知识传承：** 即使换人接手项目，也能理解历史决策的逻辑
4. **错误预防：** 记录"为什么不用方案A"，避免未来重犯同样的判断

## 15. 每日清单（Daily Checklist）

AI 辅助开发需要结构化的日常操作规范，确保每次工作都有明确的开始、过程和结束。Daily Checklist 是保证开发质量和个人效率的基础工具。

### 15.1 工作开始清单（每次新对话开始时执行）

```
□ 读取永久层文件（AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md）
□ 确认当前项目状态（查看 PROJECT_CONTEXT.md 中的"已开发模块"和"里程碑"）
□ 检查上次维护日期（周/月/季），如超过阈值则提醒用户
□ 定位当前工作模块（查看最后开发的模块或里程碑）
□ 如距离上次开发 >7天，执行跨月恢复协议（PROJECT_CONTEXT.md §9.6）
□ 向用户展示"当前项目状态摘要"，确认理解无误
□ 明确本次对话的唯一目标（单一职责原则）
```

### 15.2 工作结束清单（每次对话结束前执行）

```
□ 更新当前模块的 DEVLOG/ 日志（记录今日完成内容和问题）
□ 如模块已完成，生成 CONTINUE/ 总结文件
□ 更新 PROJECT_CONTEXT.md 中的模块状态（未开始/开发中/已完成）
□ 如有重要决策，生成决策日志到 CONTINUE/decision-log/
□ 检查上下文使用率，如 >80% 则开启新对话并保存进度
□ 向用户展示"本次工作完成摘要"，确认状态正确
```

### 15.3 每周检查清单（与 §12.1 周维护联动）

```
□ 确认本周每天都有开发日志（DEVLOG/）
□ 检查是否有模块长期处于"开发中"状态（超过14天）
□ 确认最近完成的模块已生成 CONTINUE/ 总结
□ 检查 PROJECT_CONTEXT.md 中的里程碑状态是否与实际进度一致
□ 生成周总结文件 `DEVLOG/weekly-YYYY-WXX.md`
□ 检查是否有未记录的决策（生成决策日志）
□ 确认上次周维护日期，如超过7天则提醒用户
```

### 15.4 每月检查清单（与 §12.2 月维护联动）

```
□ 清理临时层：将超过30天的 DevLog 标记为可弃
□ 检查是否有模块长期处于"未开始"状态（超过30天）
□ 检查 MODEL_CONFIG.md 中的模型配置是否仍适用
□ 生成月度摘要 `ARCHIVE/monthly-YYYY-MM.md`
□ 更新 PROJECT_CONTEXT.md 中的里程碑时间线
□ 检查所有未回顾的决策日志（14.5 决策回顾机制）
□ 确认上次月维护日期，如超过30天则提醒用户
```

### 15.5 TRAE 自动化触发

| 触发条件 | 自动执行动作 |
|----------|-------------|
| 新对话开始 | 读取永久层 + 检查维护日期 + 展示项目状态摘要 |
| 模块完成 | 自动生成 CONTINUE/ 总结 + 更新 PROJECT_CONTEXT.md |
| 上下文使用率 >70% | 提醒用户是否有关键决策未记录 + 建议开启新对话 |
| 上下文使用率 >80% | 自动触发上下文溢出处理 + 建议开启新对话 |
| 用户说"今天先这样"/"下次继续" | 自动执行工作结束清单 |
| 连续对话 >20轮 | 提醒用户是否生成了足够的总结/日志 |
| 距离上次周维护 >7天 | 在每次对话开始时提醒用户 |
| 距离上次月维护 >30天 | 在每次对话开始时提醒用户 |

### 15.6 快速指令对照表

| 场景 | 触发指令 |
|------|----------|
| 开始新工作 | "开始今天的开发，当前模块是 [模块名]" |
| 结束当天工作 | "今天先这样，保存进度" 或 "执行工作结束清单" |
| 检查项目状态 | "当前项目的开发进度如何？" |
| 执行周维护 | "执行本周项目维护" |
| 执行月维护 | "执行本月项目归档" |
