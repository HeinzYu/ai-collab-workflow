# TEMPLATE_GUIDE.md — 模板使用指南

> 本文件说明如何使用此模板框架进行 Apple App 项目的持续开发。适用于人类开发者和 TRAE Agent。

---

## 1. 快速开始

### 1.1 初始化项目

在 TRAE 中发送以下指令之一：

**完整商业项目（推荐）：**
> "这是一个完整的 Apple App 项目，包含登录、首页、设置等功能"

TRAE 将：
1. 在当前目录创建完整的模板框架结构
2. 生成所有核心文件（始终详细）
3. 生成所有模块的占位符文件

**小型工具项目：**
> "这是一个简单的工具 App，只需要登录和首页功能"

TRAE 将：
1. 生成核心文件 + 选定的模块占位符
2. 跳过 TESTING/、DEPLOYMENT/ 等不必要模块

**最小可行项目（MVP）：**
> "这是一个原型验证，只需要一个页面"

TRAE 将：
1. 仅生成核心文件 + PRD/ 下对应功能的占位符

### 1.2 验证初始化结果

项目初始化后，检查以下文件是否存在：
- `AGENTS.md` — TRAE 交互规则（始终存在）
- `PROJECT_CONTEXT.md` — 项目信息（需人类填写）
- `MODEL_CONFIG.md` — 本地模型配置（需根据实际部署填充）
- `REFERENCES.md` — 大文件索引（按需添加条目）
- `TEMPLATE_GUIDE.md` — 本文件

### 1.3 快速指令速查

| 你想做什么 | 对 TRAE 说什么 |
|-----------|---------------|
| **初始化新项目** | "这是一个完整的 Apple App 项目，包含登录、首页、设置等功能" |
| **初始化小型项目** | "这是一个简单的工具 App，只需要登录和首页功能" |
| **初始化 MVP** | "这是一个原型验证，只需要一个页面" |
| **展开模块文档** | "生成 [模块名] 功能的详细 PRD" / "完善 [模块名] 文档" |
| **生成模块总结** | "生成 [模块名] 功能的总结" |
| **继续已有功能** | "继续开发 [模块名]，参考之前的总结" |
| **添加大文件** | "将这份文档添加到项目参考中"（附上 doc/PDF/Figma） |
| **查询项目状态** | "当前项目的开发进度如何？" |
| **执行周维护** | "执行本周项目维护" |
| **执行月归档** | "执行本月项目归档" |
| **执行季审计** | "执行季度项目审计" |

## 2. 日常开发流程

### 2.1 启动新对话

每次开启新的 TRAE 对话时：
1. TRAE 自动读取核心文件（AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md）
2. 确认当前项目状态（scope、开发阶段）
3. 加载当前模块的 PRD 和相关总结

### 2.2 开发功能模块

**步骤：**
1. 在 TRAE 中说明要开发的模块："开发登录功能"
2. TRAE 检查 `PRD/login.md` 是否存在：
   - 若为占位符 → 先展开为详细文档，再开始开发
   - 若已存在 → 直接读取并开发
3. TRAE 在 `DEVLOG/` 中记录当日开发活动
4. 开发完成后，生成 `CONTINUE/login-summary.md`

### 2.3 继续已有功能

当需要继续之前未完成的功能时：
1. 在 TRAE 中说明："继续开发登录功能"
2. TRAE 自动读取：
   - `PRD/login.md` — 需求文档
   - `CONTINUE/login-summary.md` — 之前的总结
   - `DEVLOG/` 中最近的日志

### 2.4 执行周期性维护

当项目运行超过1个月时，需要定期执行维护以保持 AI 对项目上下文的理解：

**每周维护（推荐每7天）：**
> "执行本周项目维护"

TRAE 将：
1. 检查 DEVLOG/ 中是否有连续的开发记录
2. 标记长期处于"开发中"状态的模块（超过14天）
3. 确认最近完成的模块已生成 CONTINUE/ 总结
4. 生成周总结 `DEVLOG/weekly-YYYY-WXX.md`

**每月维护（推荐每30天）：**
> "执行本月项目归档"

TRAE 将：
1. 扫描并标记超过30天的 DevLog 为可弃层
2. 检查长期未开始的模块，评估是否需要调整优先级
3. 生成月度摘要 `ARCHIVE/monthly-YYYY-MM.md`
4. 更新 PROJECT_CONTEXT.md 中的维护追踪日期

**每季度维护（推荐每90天）：**
> "执行季度项目审计"

TRAE 将：
1. 清理超过90天的可弃层 DevLog（物理删除或归档）
2. 评估项目 scope 是否仍然合理（full/small/mvp）
3. 检查 MODEL_CONFIG.md 是否需要更新
4. 生成季度审计报告 `ARCHIVE/quarterly-YYYY-QX.md`

## 3. 模块展开指南

### 3.1 何时展开

| 场景 | 是否展开 |
|------|----------|
| 初始化项目时 | 否（保持占位符） |
| 用户要求"生成 XXX 的详细文档" | 是 |
| 模块开发完成时 | 是（自动生成总结） |
| 项目进入特定阶段（测试/部署） | 是 |

### 3.2 展开指令示例

| 要展开的模块 | TRAE 指令 |
|-------------|-----------|
| 登录功能 PRD | "生成登录功能的详细 PRD" |
| 技术架构 | "完善技术架构文档" |
| 测试策略 | "完善测试策略" |
| 发布流程 | "完善发布流程文档" |

### 3.3 展开后的操作

1. TRAE 将占位符文件替换为完整章节结构
2. 每个展开的文件包含填写指南
3. TRAE 在开发过程中逐步填充内容

## 4. 大文件处理指南

### 4.1 提交 doc/PDF 文档

在 TRAE 中：
1. 通过文件附件或路径提交大文件
2. 说明文档用途："将这份产品需求文档添加到项目参考中"

TRAE 自动：
1. 生成 Markdown 摘要（≤500字）
2. 将摘要写入 `REFERENCES.md`
3. 原文存储在 `ASSETS/documents/`

### 4.2 提交 Figma 设计稿

在 TRAE 中：
1. 提供 Figma 文件 URL 或 Token
2. 说明设计稿用途："这是登录页面的设计稿"

TRAE 自动：
1. 通过 Figma MCP 获取设计数据
2. 按页面/组件拆分导出
3. 在 `REFERENCES.md` 中建立索引

## 5. 项目状态管理

### 5.1 查看当前进度

在 TRAE 中询问："当前项目的开发进度如何？"

TRAE 将：
1. 读取 `PROJECT_CONTEXT.md` 中的模块状态
2. 检查各 PRD 文件是否已展开
3. 检查 DEVLOG/ 中最近的开发活动

### 5.2 更新项目信息

当项目发生重大变更时（如功能调整、技术栈更换）：
1. 在 TRAE 中说明变更内容
2. TRAE 自动更新 `PROJECT_CONTEXT.md` 中的相关信息

## 6. 常见问题

### Q: 如何判断项目是"小型"还是"完整"？

**A:** 根据功能模块数量判断：
- 1-2个核心功能 → 小型（small）
- 3-5个功能模块 → 完整（full）
- 仅1个页面验证概念 → MVP

### Q: 占位符文件怎么使用？

**A:** 占位符是简短的说明文档，包含：
- 文件目标（1-2句话）
- 章节列表（待填充）
- 填写指南

当需要详细文档时，通过 TRAE 指令展开。

### Q: 如何迁移到另一个项目？

**A:** 复制整个模板框架目录到新项目的根目录，然后：
1. 更新 `PROJECT_CONTEXT.md` 中的项目信息
2. 根据新项目需求调整模块配置

### Q: 本地模型不可用怎么办？

**A:** TRAE 会自动：
1. 重试连接（最多3次）
2. 记录错误到 `DEVLOG/`
3. 通知用户当前状态

### Q: 项目超过1个月后，AI 还能理解之前的开发吗？

**A:** 可以。通过以下机制防止"记忆衰减"：
- **月度摘要**（`ARCHIVE/monthly-YYYY-MM.md`）：每月自动生成，记录功能完成、技术决策和问题
- **模块总结**（`CONTINUE/*-summary.md`）：完成模块后立即生成，是跨会话恢复的关键
- **季度审计**（`ARCHIVE/quarterly-YYYY-QX.md`）：每90天执行，恢复项目上下文
- 详见 `PROJECT_CONTEXT.md` §9 "长期项目记忆衰减策略"

### Q: 什么时候应该切换模型？

**A:** TRAE 根据以下决策树自动选择：
- **简单任务**（文本格式化/问答）→ Gemma (26B)
- **中等任务**（代码生成 <500行）→ Gemma (26B)
- **高复杂度**（代码生成 >500行/长文档）→ Qwen (35B)
- **极高复杂度**（全架构设计）→ 评估是否切云端模型
- 详见 `MODEL_CONFIG.md` §7 "模型切换决策树"

## 7. 最佳实践

1. **每次开发前阅读核心文件** — 确保 TRAE 和人类对项目的理解一致
2. **及时生成模块总结** — 完成功能后立即生成 `CONTINUE/` 总结
3. **定期更新开发日志** — 每天至少记录一次 `DEVLOG/`
4. **大文件及时索引** — 提交大文件后立即让 TRAE 生成摘要
5. **按需展开模块** — 不要一次性展开所有模块，按开发进度逐步展开
6. **定期执行周期维护** — 超过1个月的项目，每周/月/季度执行维护（见 §2.4）
7. **防止记忆衰减** — 每月生成月度摘要，确保跨会话恢复时有足够上下文（详见 `PROJECT_CONTEXT.md` §9）
