# ai-collab-workflow 启动语使用指南

> **基于**: `ai-collab-workflow` v1.9 优化方案
> **制定日期**: 2026-07-02
> **适用范围**: 所有用户（从初学者到高级开发者）

---

## 📖 使用说明

本指南列出了 `ai-collab-workflow` 技能的所有启动语，按**场景分类**。所有指令均采用**自然语言对话风格**，无需记忆任何特殊语法或命令行参数。

**核心原则**: 像日常对话一样表达你的需求即可。

---

## §1 项目初始化类启动语

> **适用场景**: 全新项目的初始设置，根据项目规模和类型选择不同模式。

### 1.1 Full 模式（完整商业项目）

**触发语**:
- "这是一个完整的 Apple App 项目，包含 [功能]...等功能"
- "这是一个 Web 项目，包含 [功能]...等功能"

**适用场景**: 全新商业级应用（iOS/macOS/Web），包含多个功能模块。

**系统动作**:
1. 自动检测目录状态（空目录 → Full 模式；已有代码 → 降级为 Merge 模式）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 生成完整的目录结构（AGENTS.md、PROJECT_CONTEXT.md、MODEL_CONFIG.md 等）
4. 初始化所有模块占位符（PRD/、ARCHITECTURE/、TESTING/、DEPLOYMENT/ 等）
5. 初始化所有补充章节（S01-S07）

**示例**:
```
"这是一个完整的 Apple App 项目，包含登录、首页、设置等功能"
"这是一个 Web 项目，包含用户系统、商品管理、支付功能等"
```

---

### 1.2 Small 模式（小型工具项目）

**触发语**:
- "这是一个简单的工具 App，只需要 [功能1] 和 [功能2]"

**适用场景**: 小型工具类应用，只需核心模块。

**系统动作**:
1. 自动检测目录状态（空目录 → Small 模式；已有代码 → 降级为 Merge 模式）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 生成核心文件 + 选中的模块占位符
4. 跳过 TESTING/、DEPLOYMENT/（简化流程）
5. 在 PROJECT_CONTEXT.md 中标记 `scope: "small"`

**示例**:
```
"这是一个简单的工具 App，只需要登录和首页功能"
"这是一个小型管理后台，只需要用户管理和数据导出功能"
```

---

### 1.3 MVP 模式（最小可行产品）

**触发语**:
- "这是一个原型验证，只需要 [一个功能]"

**适用场景**: 最小可行产品验证，只生成对应 PRD 占位符。

**系统动作**:
1. 自动检测目录状态（空目录 → MVP 模式；已有代码 → 降级为 Merge 模式）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 仅生成核心文件 + 对应的 PRD 占位符
4. 跳过所有其他模块
5. 在 PROJECT_CONTEXT.md 中标记 `scope: "mvp"`

**示例**:
```
"这是一个原型验证，只需要一个页面"
"这是一个 MVP 项目，只需要用户注册功能"
```

## §2 已有项目集成类启动语

> **适用场景**: 已有代码库，需嵌入标准文档框架。系统会自动检测目录状态并降级为合适模式。

### 2.1 Merge 模式（标准集成）

**触发语**:
- "帮我给这个项目加上协同工作流"
- "为这个项目初始化协作开发模板"

**适用场景**: 已有代码库，需嵌入标准文档框架。

**系统动作**:
1. 运行全局前置检测（检查目录中是否有现有代码）
2. 运行交互式模型对话（自动检测物理内存 + 本地模型服务，用户确认）
3. 列出已有文件清单（inventory）
4. 展示合并计划（新增哪些文件、如何处理已有文件）
5. 等待用户确认后执行

**输出示例**:
```
检测到这是一个已有项目。在开始工作前，已与您确认模型信息（见 MODEL_CONFIG.md §0）。
我建议以下操作：
- 保留现有源码文件（不修改）
- 新增以下框架文件：AGENTS.md, PROJECT_CONTEXT.md, ...
- 以下文件已存在，建议：[跳过 / 合并 / 备份]
- 请问如何继续？
```

---

### 2.2 Shell Mode（外壳模式，v1.9 新增）

**触发语**:
- "帮我给这个项目加上协同工作流" *(系统自动检测后提示)*

**适用场景**: 项目已有功能等价文档（如 `AI_CONTEXT_MAP.md`、`AI_PROGRESS_MAP.md`），只需补充流程层文件。

**系统动作**:
1. 运行全局前置检测 + 交互式模型对话（同 Merge 模式）
2. **功能等价检测**：扫描项目自有文档，判断是否与标准文件功能等价
3. 如果检测到 ≥2 个功能等价物 → 自动进入 Shell Mode

**输出示例**:
```
检测到此项目已有完善的文档体系：
  - AI_CONTEXT_MAP.md → 替代 PROJECT_CONTEXT.md（任务路由 14 类 + 行号精度）
  - AI_PROGRESS_MAP.md → 替代进度追踪（23+ 模块 + N1-N35 编号）
  - 技术文件框架/ → 替代 ARCHITECTURE/（E2-E6 工程规格）

我建议用"外壳模式"，只补充流程层文件：
  ✓ 新增: AGENTS.md, MODEL_CONFIG.md, REFERENCES.md, DEVLOG/, CONTINUE/
  ✗ 跳过: PROJECT_CONTEXT.md, PRD/, ARCHITECTURE/, TESTING/, DEPLOYMENT/

请问是否接受？
```

**功能等价映射表**:

| 标准文件 | 项目自有文档（功能等价物） | 检测信号 |
|---------|------------------------|---------|
| `PROJECT_CONTEXT.md` | `AI_CONTEXT_MAP.md` | 包含 "任务路由" + P0/P1/P2 标注 |
| `PROJECT_CONTEXT.md`（进度部分） | `AI_PROGRESS_MAP.md` | 包含模块完成度百分比 + N1-Nxx 编号 |
| `ARCHITECTURE/` | `技术文件框架/` | 包含 E2/E3/E4/E5/E6 命名的工程规格文件 |
| `PRD/` | `PRD_` 前缀的文件 | 包含功能需求描述 |
| `TESTING/` | 代码中的测试文件 | 不创建新目录，在 AGENTS.md 中标注 |

## §3 模块开发管理类启动语

> **适用场景**: 日常功能开发与续接，适用于已有项目的持续开发阶段。
> **状态**: ✅ 已在 SKILL.md v1.9.2 (2026-07-02) 中启用

### 3.1 生成模块详细 PRD ✅ (已启用)

**触发语**:
- "生成 [模块名] 功能的详细 PRD"
- "为 [模块名] 创建需求文档"

**适用场景**: 已有项目中需补充特定功能的需求文档。

**系统动作**:
1. 检查 `PRD/` 目录是否存在；如不存在则创建占位符结构
2. 在 `PRD/<module-name>.md` 中生成（或更新）需求文档：
   - 功能描述（2-3句话）
   - 用户故事 / 验收标准
   - 相关文件路径（如果已有代码）
3. 更新 `PROJECT_CONTEXT.md` 模块列表，添加新条目（状态: "未开始"）
4. 返回确认信息

**输出示例**:
```
已生成模块 PRD: PRD/<module-name>.md

内容概要:
- 功能描述: [2-3句话]
- 用户故事: [user story format]
- 验收标准: [检查清单]

模块状态已更新到 PROJECT_CONTEXT.md (状态: 未开始).
如需开始开发此模块，请说 "开始今天的开发，当前模块是 [模块名]"。
```

---

### 3.2 生成模块总结（CONTINUE/）✅ (已启用)

**触发语**:
- "生成 [模块名] 功能的总结"
- "[模块名] 开发完成了，做个总结"

**适用场景**: 模块开发完成后，自动生成总结文档。

**系统动作**:
1. 检查 `PRD/<module-name>.md` 是否存在（需求来源）
2. 读取该模块开发期间的 `DEVLOG/` 日志
3. 生成（或更新） `CONTINUE/<module-name>-summary.md`：
   - 功能概述（2-3句话）
   - 关键技术决策
   - 已知问题和待办事项
   - 相关文件路径索引
4. 更新 `PROJECT_CONTEXT.md` 模块状态为 "已完成"（含完成日期）

**输出示例**:
```
已生成模块总结: CONTINUE/<module-name>-summary.md

内容概要:
- 功能概述: [2-3句话]
- 关键技术决策: [列表]
- 已知问题: [list or "无"]
- 相关文件索引: [文件路径]

模块状态已更新到 PROJECT_CONTEXT.md (状态: 已完成, 完成日期: YYYY-MM-DD).
```

---

### 3.3 续接已有功能 ✅ (已启用)

**触发语**:
- "继续开发 [模块名]，参考之前的总结"
- "[模块名] 上次做到一半，继续开发"

**适用场景**: 中断后继续开发，自动加载相关 CONTINUE/ 摘要和决策日志。

**系统动作**:
1. 查找 `CONTINUE/<module-name>-summary.md`（如果存在，加载它）
2. 查找 `CONTINUE/decision-log/` 中相关的决策日志（按日期排序，最近5条）
3. 读取 `PRD/<module-name>.md` 获取需求（如果存在）
4. 向用户展示 "当前模块状态"，包括：
   - 上次完成的子任务（来自总结）
   - 已知问题 / 待办事项（来自总结）
   - 相关决策日志（如果有）
5. 与用户确认后再开始开发

**输出示例**:
```
正在续接模块: [module-name]

已加载上下文:
- 需求文档: PRD/<module-name>.md (如果存在)
- 模块总结: CONTINUE/<module-name>-summary.md
  - 上次完成子任务: [description]
  - 已知问题: [list or "无"]
- 决策日志 (最近 5 条):
  - [YYYY-MM-DD] [title]: [brief description]

当前模块状态: PROJECT_CONTEXT.md 中标记为 "[status]"。
请确认是否从上次中断处继续开发？
```

---

### 3.4 每日开发启动 ✅ (已启用)

**触发语**:
- "开始今天的开发，当前模块是 [模块名]"
- "今天做 [模块名] 功能"

**适用场景**: 每日开发开始时指定当前工作模块。TRAE 加载永久层文件 + 对应 PRD 和 CONTINUE/ 摘要，然后显示项目状态摘要。

**系统动作**:
1. 加载永久层文件（AGENTS.md、MODEL_CONFIG.md、动态检测的项目自有文档）
2. 加载对应模块的 PRD (`PRD/<module-name>.md`)（如果存在）
3. 加载对应模块的 CONTINUE/ 摘要（如果该模块之前有开发记录）
4. 显示 "当前项目状态摘要"，包括：
   - 模块状态（来自 PROJECT_CONTEXT.md）
   - 相关依赖模块
   - 任何逾期项目或风险
5. 确认单职责：一次对话 = 一个模块

**输出示例**:
```
今日开发已启动 — 模块: [module-name]

已加载上下文:
- 永久层文件: AGENTS.md, MODEL_CONFIG.md [± 项目自有文档]
- 需求文档: PRD/<module-name>.md (如果存在)
- 模块总结: CONTINUE/<module-name>-summary.md (如果存在)

当前项目状态:
- 模块总览: [X] 个模块, [Y] 已完成, [Z] 开发中
- 当前模块状态: PROJECT_CONTEXT.md 中标记为 "[status]"

今日目标 (单职责): 完成 [module-name] 的 [具体子任务].
请确认开始开发，或补充具体子任务目标。
```

## §4 项目查询与管理类启动语

> **适用场景**: 查看项目进度、健康状态和整体情况。
> **状态**: ✅ 已在 SKILL.md v1.9.2 (2026-07-02) 中启用

### 4.1 查询开发进度 ✅ (已启用)

**触发语**:
- "当前项目的开发进度如何？"
- "项目现在进行到哪一步了？"

**适用场景**: 查看 PROJECT_CONTEXT.md 中的模块完成度、健康仪表板。

**系统动作**:
1. 读取 `PROJECT_CONTEXT.md` 模块状态 section (通常 §2 或 §3)
2. 显示各功能模块的完成度百分比
3. 展示整体项目健康状态

**输出示例**:
```
当前项目开发进度：
━━━━━━━━━━━━━━━━━━━━━━━
总模块数: [X]

模块状态:
- 用户登录模块: ✅ 已完成 (100%) — 完成日期: YYYY-MM-DD
- 首页展示模块: 🔄 开发中 (75%) — 预计完成: YYYY-MM-DD
- 支付功能模块: ⏸️ 未开始 (0%) — 计划启动: YYYY-MM-DD

整体进度: [Y]%
健康状态: 🟢 正常 / 🟡 关注 / 🔴 危险

最近完成模块: [module-name] (YYYY-MM-DD)
当前开发中模块: [module-name] ([Z]%)
```

**Fallback:** 如果 `PROJECT_CONTEXT.md` 不存在或无模块状态数据，TRAE 会提示用户先运行初始化命令。

### 4.2 显示项目健康仪表板 ✅ (已启用)

**触发语**:
- "显示当前项目健康仪表板"
- "查看项目整体健康状况"

**适用场景**: 查看整体项目健康状况、风险告警、系统完整性。

**系统动作**:
1. 读取 `PROJECT_CONTEXT.md` — 提取所有仪表板章节 (A-D)
2. 如果项目类型是游戏，同时读取游戏特有指标 (Section E)（如果有）
3. 生成格式化的仪表板输出

**标准项目输出示例**:
```
项目健康仪表板 — [Project Name]
━━━━━━━━━━━━━━━━━━━━━━━

A. 维护状态:
   - 上次周维护: [YYYY-MM-DD] — ✅ 正常 / ⚠️ 逾期
   - 上次月归档: [YYYY-MM-DD] — ✅ 正常 / ⚠️ 逾期
   - 上次季度审计: [YYYY-MM-DD] — ✅ 正常 / ⚠️ 逾期

B. 开发进度:
   - 总模块数: [X]
   - 已完成: [Y] ([Z]%)
   - 开发中: [A]
   - 未开始: [B] ([C]%)

C. 风险告警:
   - 逾期模块: [list or "无"]
   - 记忆衰减警告: [list or "无"]
   - 技术风险: [list or "无"]

D. 系统健康:
   - 核心文件完整性: AGENTS.md ✓ / PROJECT_CONTEXT.md ✓ / MODEL_CONFIG.md ✓
   - 决策日志数量: [X] (近30天)
   - Token 使用趋势: 📈 上升 / ➡️ 稳定 / 📉 下降 (近4周)
```

**游戏项目额外输出 (Section E)**: 当 `$game_health` 为 `true` 时，追加：
```
E. 游戏特有指标:
   - 编译状态: [0 errors / X warnings]
   - Tick 帧率抖动: [±X% deviation]
   - 确定性回放通过率: [X%]

游戏健康评分: [Excellent/Good/Warning/Danger]
```

## §5 项目维护管理类启动语

> **适用场景**: 定期文档维护与归档，适用于项目运行一段时间后的常规维护。
> **状态**: ✅ 已在 SKILL.md v1.9.2 (2026-07-02) 中启用

### 5.1 执行周维护 ✅ (已启用)

**触发语**:
- "执行本周项目维护"

**适用场景**: 每周自动检查 DEVLOG/、更新进度、生成周报。

**系统动作**:
1. 检查最近 7 天的 DEVLOG/ 日志
2. 更新 PROJECT_CONTEXT.md 中的模块进度
3. 生成周报摘要（写入 DEVLOG/weekly-YYYY-Www.md）

**输出示例**:
```
本周项目维护完成:
- 新增开发日志: 5 条
- 更新的模块: 首页展示 (75% → 80%)
- 风险告警: 1 条（已处理）

周报已生成: DEVLOG/weekly-2026-W27.md
```

---

### 5.2 执行月归档 ✅ (已启用)

**触发语**:
- "执行本月项目归档"

**适用场景**: 生成月度总结，归档到 SUPPLEMENTARY/ 和 ARCHIVE/。

**系统动作**:
1. 检查最近 30 天的所有日志和文档变更
2. 生成月度总结报告（写入 SUPPLEMENTARY/monthly-YYYY-MM.md）
3. 更新 ARCHIVE/ 目录

**输出示例**:
```
本月项目归档完成:
- 总开发日志: 20 条
- 完成的模块: 用户登录 (100%)
- 进行中的模块: 首页展示 (80%)

月报已生成: SUPPLEMENTARY/monthly-2026-07.md
```

---

### 5.3 执行季度审计 ✅ (已启用)

**触发语**:
- "执行季度项目审计"

**适用场景**: 全面审查项目健康度、合规评分趋势、已验证约束提取。

**系统动作**:
1. 检查最近 90 天的所有日志和文档变更
2. 全面审查项目健康度（模块进度 + 合规评分 + 风险告警）
3. 提取已验证约束（经过多次对话验证的设计决策）并写入 `CONTINUE/verified-constraints.md`
4. 生成季度审计报告

**输出示例**:
```
季度项目审计完成:
- 总开发日志: 60 条
- 完成的模块: 2/3 (用户登录 ✅, 首页展示 ✅)
- 合规评分趋势: ↑ 从 65% 提升到 78%

已验证约束提取:
- 技术选型: 使用 SwiftUI 作为前端框架 (经 5 次对话验证)
- 架构决策: 采用 MVVM 模式 (经 8 次对话验证)

审计报告已生成: SUPPLEMENTARY/quarterly-2026-Q3.md
```

## §6 会话管理类启动语

> **适用场景**: 每日开发开始与结束，确保每次对话的状态连续性。

## 🔧 附录：技术细节（开发者参考）

### A. Context Budget 自适应算法

初始化时执行以下调度算法：
1. 检测所有项目自有核心文件（AI_CONTEXT_MAP.md, AI_PROGRESS_MAP.md 等）
2. 估算每个文件的 token 消耗（按 KB × 1.5 安全系数）
3. 根据 T1-T5 档位确定永久层预算上限
4. 按优先级（P0 > P1 > P2）填充永久层，直到预算耗尽
5. 超出预算的文件自动降级为 "Stable Layer"（按需加载，非永久层）

### B. Shell Mode 功能等价检测

当检测到 ≥2 个功能等价物时，自动进入 Shell Mode：
- 仅创建流程层文件（AGENTS.md, MODEL_CONFIG.md, REFERENCES.md, DEVLOG/, CONTINUE/）
- 跳过标准框架文件（PROJECT_CONTEXT.md, PRD/, ARCHITECTURE/ 等）
- AGENTS.md §3 (Always Load) 动态列出项目自有文件

### C. 合规检查机制 (v1.9.2 更新)

- **纯 Markdown 操作**: 所有合规检查通过读取项目文件（AGENTS.md, PROJECT_CONTEXT.md 等）并计算评分完成，无需外部脚本
- **四阶段检查**: 核心文件 / Context 生命周期 / 硬件自适应 / 维护机制
- **评分范围**: 0-100（核心文件40分 + 标准目录30分 + Context生命周期15分 + 硬件自适应10分 + 维护机制5分）
- **趋势追踪**: 首次检查写入 `.compliance_history`，后续每次合规检查追加新条目
- **自动修复**: 可自动生成缺失核心文件的占位符（PROJECT_CONTEXT.md, PRD/, CONTINUE/, DEVLOG/）
- **模块完成后自动检查**: 系统检测到模块完成时，自动运行合规检查并记录评分到历史
